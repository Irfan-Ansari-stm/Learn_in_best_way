14.1 Single Page Applications (SPA)

ğŸ“Œ Summary: The Modern Web Experience

A Single Page Application (SPA) is a web app that loads a single HTML page and dynamically updates that page as the user interacts with the app â€” no full page reloads.

ğŸ”¹ Traditional Multi-Page App vs SPA

FeatureTraditional WebsiteSingle Page Application
Page LoadFull page reload on every navigationNo page reload â€” only component changes
Server RequestsHTML, CSS, JS downloaded each timeHTML/CSS/JS loaded once, then API calls only
User ExperienceVisible page transitions, flickeringSmooth, app-like experience
PerformanceSlower due to repeated downloadsFaster after initial load
NavigationBrowser URL changes with full requestURL changes via JavaScript (history API)
ExamplesWordPress, traditional blogsGmail, Facebook, Twitter, Netflix

âš¡ How SPAs Work Internally


// Traditional website flow:
User clicks link â†’ Browser requests new HTML from server â†’ Server sends new page â†’ Browser renders

// SPA flow:
User clicks link â†’ JavaScript intercepts click â†’ Updates URL via history.pushState() â†’ 
React Router matches route â†’ Renders appropriate component â†’ DOM updates without reload

ğŸ§  Mental Model: SPA as a Desktop App

Think of an SPA like a desktop application:

  1. You install it once (download all assets)
  2. You open it (load index.html)
  3. You navigate between screens (components change)
  4. The app never fully closes â€” just shows different views

ğŸ’¡ Why SPAs Are Essential Today

  • Speed: No waiting for server responses after initial load
  • Responsiveness: Instant UI updates feel native
  • Offline Capabilities: Can cache resources for offline use
  • Consistent UX: Same look and feel across all pages
  • API-Centric: Perfect for modern REST/GraphQL backends
âš ï¸ Important: SPAs require proper SEO handling (server-side rendering or static generation) since search engines may not execute JavaScript.
ğŸš€ Deep Dive: History API and Client-Side Routing

SPAs rely on the HTML5 History API to manipulate browser URLs without reloading.

ğŸ” The History API Methods

// 2. Replace current state

// 4. Listen for navigation events

  console.log('Route changed to:', location.pathname);

ğŸ” How Path Matching Works Internally

React Router uses a simple algorithm:

  1. Start at the top of the Routes list
  2. Compare current URL with each Route's path
  3. When a match is found, render that component and stop
  4. If no match, render nothing (or a 404 page)

function BlogPost() {

  return (
    <div className="component-preview">
      <h2>{post?.title}</h2>
      <p>Author: {post?.author}</p>
      <p>Published: {post?.date}</p>
      <div dangerouslySetInnerHTML={{ __html: post?.content }} />
    </div>
  );
}

// ProductPage.jsx
import { useParams, useEffect, useState } from 'react';

function ProductPage() {
  const { productId, variantId } = useParams();
  const [product, setProduct] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Fetch product data
    const fetchProduct = async () => {
      try {
        let url = `/api/products/${productId}`;
        if (variantId) {
          url += `/variants/${variantId}`;
        }
        
        const response = await fetch(url);
        const data = await response.json();
        setProduct(data);
      } catch (error) {
        console.error('Failed to fetch product:', error);
      } finally {
        setLoading(false);
      }
    };

    fetchProduct();
  }, [productId, variantId]);

  if (loading) return <div>Loading product...</div>;

  return (
    <div className="component-preview">
      <h2>{product?.name}</h2>
      <p>ID: <span className="parameter-badge">{productId}</span></p>
      {variantId && (
        <p>Variant: <span className="parameter-badge">{variantId}</span></p>
      )}
      <p>Price: ${product?.price}</p>
      <button>Add to Cart</button>
    </div>
  );
}

14.5 Nested Routes

ğŸ“Œ Summary: Organizing Complex UI Hierarchies

Nested routes allow you to build complex layouts where child routes render inside parent components â€” perfect for dashboards, tabs, and hierarchical structures.

ğŸ”¹ Core Concept: Parent-Child Relationship

In nested routing:

  1. A parent route renders a layout component
  2. The layout component contains a <Outlet /> placeholder
  3. Child routes render into that outlet
  4. The URL reflects the hierarchy

âš¡ Basic Nested Route Syntax


import { Routes, Route, Outlet } from 'react-router-dom';
import DashboardLayout from './DashboardLayout';
import DashboardHome from './DashboardHome';
import Settings from './Settings';
import Profile from './Profile';

function App() {
  return (
    <Routes>
      <Route path="/" element={<Home />} />
      
      <!-- Parent route with Outlet -->
      <Route path="/dashboard" element={<DashboardLayout />} >
        <!-- Child routes will render inside DashboardLayout's Outlet -->
        <Route index element={<DashboardHome />} />
        <Route path="settings" element={<Settings />} />
        <Route path="profile" element={<Profile />} />
      </Route>
      
      <Route path="*" element={<NotFound />} />
    </Routes>
  );
}

// DashboardLayout.jsx
import { Outlet, Link } from 'react-router-dom';

        <Link to="/dashboard/settings" className="nav-link">Settings</Link>
        <Link to="/dashboard/profile" className="nav-link">Profile</Link>
      </nav>
      
      <!-- This is where child routes render -->
      <Outlet />
    </div>
  );
}

// DashboardHome.jsx

    </div>

ğŸ’¡ Why Nested Routes Matter

  • Reusable Layouts: Header, sidebar, footer stay consistent
  • URL Hierarchy: /dashboard/settings clearly shows relationship
  • Component Composition: Share logic and state between parent and children
  • Code Organization: Group related routes together

ğŸ’¡ Pro Tips for Nested Routes

  1. Always use <Outlet /> in parent components â€” it's required
  2. Use index prop for the default child route â€” renders when path matches exactly
  3. Parent routes don't need to render anything â€” they just provide layout
  4. URLs reflect the hierarchy â€” /users/123/posts shows clear relationship
ğŸ§  Mental Model: Think of nested routes like folders in a file system. The parent folder (route) contains subfolders (child routes), and the outlet is the window that displays the contents of the selected folder.
