Software Testing Deep Dive

Mastering Integration, E2E, Coverage & CI Testing

18.5 Integration Testing

Definition

Integration testing is the process of verifying that multiple software modules or components work together as expected when combined.

It occurs after unit testing and before system testing. The goal is to detect interface defects between integrated units.

Detail

Integration testing addresses problems that unit tests can't catch:

  • Incorrect data flow between modules
  • Communication protocol mismatches
  • Resource contention (files, databases, network)
  • Timing issues in concurrent systems
  • Environment-specific configuration errors

Two main approaches:

  • Top-down: Start from high-level modules and use stubs for lower-level ones.
  • Bottom-up: Start from lowest-level modules and use drivers for higher-level ones.
  • Big Bang: Integrate all components at once (risky but simple).
  • Sandwich/Hybrid: Combine top-down and bottom-up approaches.
Syntax & Tools

Common tools and patterns:

// Example using Jest + Supertest for API integration testing
const request = require('supertest');
const app = require('../app'); // Your Express app

describe('User API Integration Tests', () => {
  it('should create a user and return 201', async () => {
    const response = await request(app)
      .post('/api/users')
      .send({ name: 'Alice', email: 'alice@example.com' })
      .expect(201);

    expect(response.body.name).toBe('Alice');
    expect(response.body.email).toBe('alice@example.com');
    expect(response.body.id).toBeDefined();
  });

  it('should get all users', async () => {
    const response = await request(app)
      .get('/api/users')
      .expect(200);

    expect(Array.isArray(response.body)).toBe(true);
  });
});

Popular frameworks:

  • Java: JUnit + TestNG
  • Python: pytest, unittest
  • JavaScript: Jest, Mocha, Supertest
  • Go: testify
Example

Scenario: Online Shopping System

  1. Module A: User Authentication Service
  2. Module B: Product Catalog Service
  3. Module C: Payment Processing Service
  4. Module D: Order Management Service

Integration Test Case:

When a user logs in (A), searches for product (B), makes payment (C), then order is created (D) — does the entire flow work?

Test Steps:

1. POST /login → returns auth token
2. GET /products?category=books → returns product list
3. POST /payment → with valid card, returns success
4. POST /orders → with cart items and payment ID
5. GET /orders/{id} → should return complete order with status "confirmed"

If step 4 fails because payment service returned a different format than expected → Integration test catches it!

Uses
  • Validating communication between microservices
  • Testing database connections and queries across services
  • Ensuring third-party API integrations work correctly
  • Verifying message queues (Kafka, RabbitMQ) handle messages properly
  • Confirming caching layers (Redis) integrate correctly with business logic
  • Essential for DevOps pipelines — catches regressions early

Key Benefit: Finds bugs that occur only when components interact — often more severe than unit test failures.

18.6 End-to-End Testing with Cypress/Playwright

Definition

End-to-End (E2E) testing simulates real user scenarios by interacting with the application through its UI — just like a human would.

Tests the entire system from start to finish: browser → frontend → API → database → external services → response.

Detail

E2E tests are the closest to real-world usage:

  • Run in actual browsers (Chrome, Firefox, Safari)
  • Interact with DOM elements, clicks, forms, navigation
  • Wait for asynchronous operations (API calls, animations)
  • Can test responsive design and accessibility

Cypress vs Playwright:

Feature Cypress Playwright
Browser Support Chrome, Firefox, Edge Chromium, Firefox, WebKit
Auto-waiting Yes Yes
Network Mocking Native Advanced
Parallel Testing Requires Cloud Native support
Mobile Testing No Partial via device emulation
Debugging Excellent dev tools Excellent dev tools
Syntax & Tools

Cypress Example:

// cypress/e2e/login_spec.js
describe('User Login Flow', () => {
  it('logs in successfully and sees dashboard', () => {
    cy.visit('/login')
    
    cy.get('[data-cy=email]').type('user@test.com')
    cy.get('[data-cy=password]').type('password123')
    cy.get('[data-cy=submit-btn]').click()
    
    // Auto-waits for redirect
    cy.url().should('include', '/dashboard')
    cy.contains('Welcome, User!').should('be.visible')
  })
})

Playwright Example:

// playwright/test/login.test.js
import { test, expect } from '@playwright/test';

test('should log in successfully', async ({ page }) => {
  await page.goto('/login');
  
  await page.fill('[data-cy=email]', 'user@test.com');
  await page.fill('[data-cy=password]', 'password123');
  await page.click('[data-cy=submit-btn]');
  
  await expect(page).toHaveURL(/\/dashboard/);
  await expect(page.locator('text=Welcome, User!')).toBeVisible();
});
Example

Scenario: Amazon-like e-commerce checkout

  1. User navigates to homepage
  2. Searches for "wireless headphones"
  3. Selects first result
  4. Adds to cart
  5. Proceeds to checkout
  6. Enters shipping address
  7. Selects payment method
  8. Confirms order
  9. Sees confirmation page with order number

Why this matters: Even if every component works alone, a missing CORS header, misconfigured CDN, or race condition in the cart service might break the entire flow.

This test runs in a real browser — just like your customer would experience it.

Uses
  • Validate critical user journeys (signup, purchase, login)
  • Ensure UX consistency across devices and browsers
  • Prevent regression in core business workflows
  • Verify third-party integrations (payment gateways, analytics)
  • Smoke testing before production deployments
  • Regression suite for QA teams

Best Practice: Use E2E tests sparingly — they're slow and flaky. Focus on 5-10 critical paths. Complement with unit and integration tests.

18.7 Test Coverage and Reporting

Definition

Test coverage measures how much of your source code is executed during testing.

It's a quantitative metric that helps identify untested parts of your codebase.

Detail

Types of coverage metrics:

  • Statement Coverage: Percentage of executable statements run
  • Branch Coverage: Percentage of decision branches (if/else, switch) taken
  • Function Coverage: Percentage of functions called
  • Line Coverage: Similar to statement, but at line level
  • Condition Coverage: Each boolean sub-expression evaluated

Important Note: High coverage ≠ good tests. You can have 100% coverage with trivial tests that don't validate behavior!

Goal: Aim for 70-90% branch coverage on critical modules. Don't chase 100% — focus on meaningful tests.

Syntax & Tools

Jest + Istanbul (JavaScript):

# Add to package.json
{
  "scripts": {
    "test:coverage": "jest --coverage"
  }
}

Python (pytest + coverage.py):

pip install pytest-cov
pytest --cov=myapp --cov-report=html

Java (JaCoCo):


  org.jacoco
  jacoco-maven-plugin
  0.8.8
  
    
      
        prepare-agent
        report
Example

Code Under Test:

// calculator.js
function divide(a, b) {
  if (b === 0) {
    throw new Error('Cannot divide by zero');
  }
  return a / b;
}

Test Suite:

// calculator.test.js
test('divides two numbers correctly', () => {
  expect(divide(10, 2)).toBe(5);
});

test('throws error when dividing by zero', () => {
  expect(() => divide(5, 0)).toThrow('Cannot divide by zero');
});

Result:

  • Statement Coverage: 100% (both branches covered)
  • Branch Coverage: 100% (true and false paths tested)
  • Function Coverage: 100%

Report Output (HTML):

In practice, reports show which lines are missed — guiding developers to write better tests.

Uses
  • Identify untested code paths before release
  • Improve code quality by reducing blind spots
  • Meet compliance requirements (medical, finance software)
  • Track testing progress over time
  • Set quality gates in CI/CD pipelines (e.g., fail build if coverage drops below 80%)
  • Help new developers understand what’s important to test

Pro Tip: Combine coverage reports with code reviews. Ask: "Why is this line not tested?" instead of just "What's missing?"

18.8 Continuous Integration Testing

Definition

Continuous Integration (CI) is the practice of automatically building, testing, and validating code changes as soon as they are committed to version control.

The goal: Detect integration issues early, reduce risk, and deliver software faster with confidence.

Detail

CI is not just running tests — it's a workflow:

  1. Developer pushes code to Git (GitHub, GitLab, Bitbucket)
  2. CI server detects change → triggers pipeline
  3. Pipeline runs: lint → unit tests → integration tests → build → E2E tests → coverage → deploy to staging
  4. Results reported immediately to developer
  5. If any step fails → build marked red → team notified

Benefits:

  • Fast feedback loop (minutes instead of days)
  • Reduced merge conflicts
  • Consistent environment (no "works on my machine")
  • Automated documentation and artifact generation
  • Enforces coding standards and security scans
Syntax & Tools

GitHub Actions (.github/workflows/ci.yml):

name: CI Pipeline
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Run Lint
        run: npm run lint
        
      - name: Run Unit Tests
        run: npm test
        
      - name: Run Integration Tests
        run: npm run test:integration
        
      - name: Generate Coverage Report
        run: npm run test:coverage
        
      - name: Upload Coverage to Codecov
        uses: codecov/codecov-action@v3
        with:
          file: ./coverage/lcov.info
          
      - name: Build App
        run: npm run build
        
      - name: Deploy to Staging
        uses: appleboy/heroku-deploy@v3.0.12
        with:
          heroku_api_key: ${{ secrets.HEROKU_API_KEY }}
          heroku_app_name: "my-app-staging"
          heroku_email: "dev@company.com"

Other CI Tools:

  • Jenkins (self-hosted, highly customizable)
  • GitLab CI/CD (built into GitLab)
  • CircleCI
  • Azure Pipelines
  • Travis CI
Example

Scenario: Team working on an e-commerce API

  1. Developer Alice adds a new feature: discount codes
  2. She writes unit tests for discount calculation logic
  3. She writes integration tests for API endpoints
  4. She commits and pushes to feature/discount-code branch
  5. GitHub Actions triggers:
    • Lint checks pass ✅
    • Unit tests: 12/12 passed ✅
    • Integration tests: 5/5 passed ✅
    • Coverage: 87% ✅
    • Docker image built ✅
    • Deployed to staging ✅
  6. Team lead reviews PR — sees green checks → merges
  7. Production deployment happens later via CD pipeline

Without CI: Alice's code sits for 3 days → merged → breaks production → blame game starts.

With CI: Issue caught in 2 minutes → fix pushed in 10 minutes.

Uses
  • Automate testing across multiple environments and OS
  • Enforce quality gates before merging code
  • Enable frequent releases (daily or multiple times per day)
  • Reduce manual QA effort
  • Provide immediate feedback to developers
  • Build reproducible artifacts for deployment
  • Integrate security scanning (SAST/DAST)
  • Support DevOps culture: "You build it, you test it, you own it"

Golden Rule: If it's not automated, it doesn't happen consistently. CI turns testing from a chore into a safety net.

Key Takeaways

  • ✅ Integration testing finds bugs between components — not within them.
  • ✅ E2E tests simulate real users — use them for critical paths only.
  • ✅ Coverage tells you what was tested — not whether it was tested well.
  • ✅ CI automates everything — from lint to deployment — making software delivery reliable.
  • ✅ These four layers form a pyramid: Unit (wide base) → Integration → E2E → CI (top automation layer).
