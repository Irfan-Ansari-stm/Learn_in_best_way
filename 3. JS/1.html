1.1 What is JavaScript?

Definition: What is JavaScript?

JavaScript (JS) is a high-level, interpreted programming language that enables dynamic, interactive behavior on websites and web applications. Originally created by Brendan Eich in 1995 while working at Netscape, JavaScript was designed to make web pages more responsive and interactive.

Unlike static HTML (which structures content) and CSS (which styles it), JavaScript adds behavior - making elements respond to user actions, manipulate data, communicate with servers, and update content without reloading the page.

Today, JavaScript runs everywhere:

  • Web browsers (client-side)
  • Servers via Node.js
  • Moblie apps (React Native, Ionic)
  • Desktop apps (Electron)
  • IoT devices
History and evolution of JavaScript

JavaScript's journey is fascinating:

  • 1995: Created by Brendan Eich in just 10 days at Netscape as "Mocha", later renamed LiveScript, then JavaScript.
  • 1996: Microsoft released JScript as a clone for Internet Explorer, leading to browser wars.
  • 1997: JavaScript standardized as ECMAScript (ES1) by ECMA International to ensure cross-browser compatibility.
  • 2009: ES5 introduced critical features like strict mode, JSON support, and array methods (map(), filter(), reduce()).
  • 2015: ES6/ES2015 revolutionized JavaScript with classes, modules, arrow functions, promises, let/const, template literals, and more.
  • 2016-Present: Annual updates (ES2016, ES2017...) introduced async/await, optional chaining, nullish coalescing, private fields, and more.
üí° Pro Tip: Think of JavaScript as a living language - new features are added yearly, making it one of the fastest-evolving languages!
JavaScript vs Java vs other languages

Despite similar names, JavaScript and Java are fundamentally different:

Feature JavaScript Java Python
Type System Dynamically typed Statically typed Dynamically typed
Execution Interpreted (by browser/Node.js) Compiled to bytecode (JVM) Interpreted
Primary Use Web development Enterprise apps, Android Data science, scripting
Object Model Prototype-based Class-based Class-based
Concurrency Event loop, single-threaded Multi-threaded Multi-threaded (with GIL)

Key Differences:

  • JavaScript: Runs in browsers, asynchronous by design, flexible syntax, functional-first approach.
  • Java: Compiled, verbose, enterprise-grade, strong typing, requires JVM.
  • Python: Readable syntax, excellent for data science, slower execution speed.
üéØ Real-world analogy: JavaScript is like a jazz improvisation - flexible and reactive. Java is like classical symphony - structured and precise.
Client-side vs Server-side JavaScript

JavaScript runs in two primary environments:

Client-Side JavaScript

Runs in the user's web browser. Responsible for:

  • DOM manipulation (changing webpage content)
  • Event handling (clicks, keypresses, form submissions)
  • Animations and visual effects
  • AJAX requests (fetching data without page reload)
// Client-side example: Change button text on click
document.getElementById('myButton').addEventListener('click', function() {
    this.textContent = 'Clicked!';
    this.style.backgroundColor = '#ff8c00';
});
Server-Side JavaScript

Runs on the server using Node.js. Responsible for:

  • Handling HTTP requests/responses
  • Database operations
  • File system access
  • API creation
  • Authentication and authorization
// Server-side example: Express.js API endpoint
const express = require('express');
const app = express();

app.get('/api/users', (req, res) => {
    res.json([{id: 1, name: 'Alice'}, {id: 2, name: 'Bob'}]);
});

app.listen(3000, () => console.log('Server running on port 3000'));
üåê Modern web apps use both! Client-side handles UI interaction, server-side handles data processing and storage.
JavaScript engines (V8, SpiderMonkey, JavaScriptCore)

JavaScript engines are programs that execute JavaScript code. Each browser has its own engine:

Engine Browser/Platform Key Features
V8 Chrome, Edge, Node.js Just-In-Time (JIT) compilation, TurboFan optimizer, extremely fast
SpiderMonkey Firefox Open-source, excellent memory management, supports WebAssembly
JavaScriptCore Safari, iOS Optimized for Apple hardware, low power consumption
Chakra Legacy Internet Explorer Discontinued after Edge transition to Chromium

How engines work:

  1. Parsing: Convert source code to Abstract Syntax Tree (AST)
  2. Compilation: Translate AST to bytecode or machine code
  3. Execution: Run the compiled code
  4. Optimization: Monitor frequently used code and optimize it
‚ö° V8's secret sauce: It compiles JavaScript directly to machine code (not bytecode), making it incredibly fast!
ECMAScript standards and versions

ECMAScript (ES) is the official specification that JavaScript implements. Here's the evolution:

Version Year Key Features Importance
ES1 1997 Basic syntax, variables, functions Foundation
ES3 1999 Regular expressions, try/catch Widespread adoption
ES5 2009 JSON, strict mode, array methods Modern JS foundation
ES6/ES2015 2015 Classes, modules, arrow functions, promises Revolutionary update
ES2016 2016 Array.includes(), exponentiation operator (**) Small but useful
ES2017 2017 async/await, Object.values(), Object.entries() Async programming made easy
ES2020 2020 Nullish coalescing (??), optional chaining (?.) Improved error handling
ES2023 2023 Array.prototype.findLast(), Array.prototype.toSorted() Modern conveniences

Current Standard: ES2024 (latest) includes:

  • Top-level await (outside async functions)
  • Private class fields and methods
  • Promise.any() improvements
  • RegExp match indices
üìå Best Practice: Use modern JavaScript (ES2020+) but transpile with Babel if supporting older browsers.

1.3 First Steps with JavaScript

Writing your first JavaScript program

Let's create your first JavaScript program - the classic "Hello World!"

Method 1: Direct in HTML
<!DOCTYPE html>
<html>
<head>
    <title>My First JS Program</title>
</head>
<body>
    <h1>Welcome to JavaScript!</h1>
    
    <script>
        // This is your first JavaScript code!
        alert('Hello World!');
        console.log('JavaScript is running!');
        document.body.style.backgroundColor = '#0a192f';
    </script>
</body>
</html>
Method 2: External JavaScript file

Create script.js:

// script.js
function greetUser(name) {
    return `Hello, ${name}! Welcome to JavaScript programming.`;
}

// Call the function
console.log(greetUser('Student'));

// Interactive element
const button = document.createElement('button');
button.textContent = 'Click Me!';
button.style.padding = '10px 20px';
button.style.backgroundColor = '#ff8c00';
button.style.color = 'white';
button.style.border = 'none';
button.style.borderRadius = '5px';
button.style.cursor = 'pointer';

button.addEventListener('click', () => {
    alert('You clicked the button!');
});

document.body.appendChild(button);

Link it in HTML:

<script src="script.js"></script>
üéØ Key Insight: JavaScript executes sequentially from top to bottom. Place scripts at bottom of body for faster page load!

Linking JavaScript to HTML

There are three ways to connect JavaScript with HTML:

1. Inline JavaScript (NOT recommended)
<button onclick="alert('Clicked!')">Click Me</button>

Problems: Hard to maintain, mixes structure with behavior, hard to debug.

2. Internal JavaScript (in head/body)
<head>
    <script>
        document.addEventListener('DOMContentLoaded', function() {
            document.querySelector('button').addEventListener('click', function() {
                alert('Button clicked!');
            });
        });
    </script>
</head>
3. External JavaScript (RECOMMENDED)
<!-- In HTML head or before closing body tag -->
<script src="js/main.js"></script>
// main.js
document.addEventListener('DOMContentLoaded', function() {
    // Wait until HTML is fully loaded
    const buttons = document.querySelectorAll('button');
    buttons.forEach(btn => {
        btn.addEventListener('click', () => {
            console.log('Button clicked!');
        });
    });
});
‚úÖ Best Practice: Always use external JavaScript files and wait for DOMContentLoaded event!
Script tags and loading strategies

Where you place your script tags matters for performance!

Three Loading Strategies:
1. Default (blocking)
<head>
    <script src="script.js"></script>
</head>

Problem: Browser stops parsing HTML until script loads/executes. Page appears blank longer.

2. defer
<head>
    <script src="script.js" defer></script>
</head>

Behavior: Script downloads while HTML parses, executes after HTML parsing complete.

Best for: Scripts that need full DOM (most common case).

3. async
<head>
    <script src="analytics.js" async></script>
</head>

Behavior: Script downloads asynchronously, executes immediately when downloaded (may interrupt HTML parsing).

Best for: Independent scripts like analytics, ads (don't depend on DOM).

Recommended Pattern:
<!DOCTYPE html>
<html>
<head>
    <meta charset="UTF-8">
    <title>My App</title>
    <!-- CSS goes here -->
    <link rel="stylesheet" href="styles.css">
</head>
<body>
    <!-- Your HTML content -->
    <h1>Welcome to My Website</h1>
    <p>This content loads quickly!</p>
    
    <!-- JavaScript at bottom with defer -->
    <script src="script.js" defer></script>
</body>
</html>
üöÄ Performance Rule: Always put scripts at the bottom with defer unless you have a specific reason not to!
Comments and code documentation

Comments make your code readable and maintainable.

Comment Types:
// Single-line comment - use for short explanations

/*
 * Multi-line comment
 * Use for detailed explanations
 * Or temporarily disabling code blocks
 */

// Documenting a function
/**
 * Calculates the sum of two numbers
 * @param {number} a - First number
 * @param {number} b - Second number
 * @returns {number} The sum of a and b
 * @example
 * add(2, 3) // returns 5
 */
function add(a, b) {
    return a + b;
}

// TODO comments for future work
// TODO: Implement error handling for negative numbers

// HACK comments for temporary solutions
// HACK: This is a workaround for browser bug #12345

// FIXME comments for bugs
// FIXME: This condition doesn't handle null values properly
Documentation Best Practices:
  • Write comments that explain WHY, not WHAT (code shows what)
  • Update comments when you update code
  • Use JSDoc format for functions
  • Don't over-comment obvious code
  • Use consistent style throughout project
üìö Remember: Good comments turn your code from "works" to "understandable by others (and future you)!"
