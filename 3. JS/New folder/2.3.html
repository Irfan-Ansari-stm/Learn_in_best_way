▸ 1.2.1 Executive Summary: What is a “JavaScript development environment”?
Use this as a mental checklist: “If I have these 5 things, I am ready to build serious JS projects.”

A development environment is simply the combination of tools + configurations + workflows that you use to write, run, debug, and share your code. For JavaScript, a practical environment typically consists of:

  • A good editor or IDE (VS Code, WebStorm) with syntax highlighting, IntelliSense, and extensions.
  • A modern browser (Chrome, Firefox, Edge) with powerful developer tools (DevTools).
  • Node.js runtime so you can execute JS outside the browser (CLI tools, servers, scripts).
  • Package manager (npm, yarn, pnpm) to install and manage third-party libraries.
  • Version control using Git so your code is backed up, trackable, and shareable.
Professional mental model

A beginner often thinks: “I just need a text editor and a browser.”
A professional thinks: “I need a reproducible environment where my code is easy to write, easy to run, easy to debug, and impossible to lose.”

▸ 1.2.2 Code editors & IDEs (VS Code, WebStorm)

You can technically write JavaScript in Notepad, but that’s like doing PhD math on tissue paper. A serious JS setup needs at least one of these:

▸ Visual Studio Code (VS Code)
  • Free, open source, cross-platform (Windows, macOS, Linux).
  • Excellent JS/TS support, built-in terminal, Git integration.
  • Huge ecosystem of extensions (ESLint, Prettier, Node debugging, etc.).

Typical beginner-to-expert VS Code setup for JavaScript:

  1. Install VS Code from official website.
  2. Install these recommended extensions:
    • ESLint – for code quality & linting.
    • Prettier – for automatic code formatting.
    • JavaScript (ES6) code snippets – for productivity.
  3. Enable “Format on Save” & “Editor: Tab Size 2 or 4” in settings based on your team.

// Example: VS Code settings snippet (JSON-like)
{
    "editor.tabSize": 2,
    "editor.formatOnSave": true,
    "files.trimTrailingWhitespace": true,
    "javascript.updateImportsOnFileMove.enabled": "always"
}
You don't memorize this for exams, but in real life such settings make your JS projects clean and consistent.
▸ WebStorm (JetBrains)
  • Paid IDE (often free for students) focused on JavaScript and web technologies.
  • Deep refactoring tools, integrated inspections, smart code analysis.
  • Strong support for complex projects (monorepos, large codebases).

If you’re working on enterprise-scale applications or doing research on large JS code structures, WebStorm can save a lot of time via auto-refactoring and navigation.

▸ Minimum requirements for any JS editor
  • Syntax highlighting and bracket matching.
  • Code completion / IntelliSense.
  • Built-in search & replace, file explorer.
  • Integration with Git (commit, diff, branch view).

For viva: “VS Code is a lightweight, extensible code editor ideal for JavaScript, while WebStorm is a full IDE with advanced refactoring and inspection capabilities.”

▸ 1.2.3 Browser developer tools mastery

Browser DevTools are your laboratory for JavaScript. Without them, debugging is pain. With them, you can inspect variables, network calls, performance, and DOM in real time.

▸ Opening DevTools
  • Keyboard shortcuts: F12 or Ctrl+Shift+I (Windows / Linux), Cmd+Option+I (macOS).
  • Right-click → “Inspect” on any web page element.

Main panels you should know:

  • Elements: Inspect and modify HTML/CSS live.
  • Console: Run JavaScript interactively and see logs.
  • Sources: View files, set breakpoints, debug JS line-by-line.
  • Network: See HTTP requests, responses, AJAX calls.
▸ Using the Console panel

The Console is like a REPL (Read–Eval–Print Loop) for JavaScript inside your browser. For early learning, you can treat it as your playground. 


// In the browser DevTools console:
1 + 2
"Hello".toUpperCase()
Math.random()
console.log("Testing my environment");
Every expression you type is evaluated by the JS engine inside the browser.

The same Console is where your console.log() calls appear when you run scripts from an HTML file.

▸ Breakpoints & step-by-step debugging

Instead of trying to understand everything via logs, you can pause the program at a specific line (a breakpoint) and watch variables in real time.

  1. Open DevTools → Sources tab.
  2. Open the JavaScript file (e.g., main.js).
  3. Click on the line number where you want to pause.
  4. Reload the page; execution will pause at that line.

Then you can:

  • Hover variables to see values.
  • Use “Step over”, “Step into”, “Step out” to control execution.
  • Inspect call stack to see which function called which.

For viva: “Browser DevTools provide runtime inspection, breakpoints, network analysis, and console access, which are essential for debugging real-world JavaScript applications.”

▸ 1.2.4 Node.js installation and setup

Node.js is a runtime that allows you to execute JavaScript outside the browser (on your machine or on servers). It uses the V8 engine and adds system-level APIs (file system, network, etc.).

▸ Why do we need Node.js for learning JS?
  • Run JS scripts directly from the terminal (no HTML needed).
  • Use npm/yarn/pnpm to install libraries and tools.
  • Build real backend servers, CLIs, automation scripts.
▸ Basic installation flow
  1. Download Node.js from the official website (LTS version recommended for stability).
  2. Install using the system-specific installer (Next → Next → …).
  3. Verify installation in terminal:

// In terminal / command prompt:
node -v
npm -v
You should see versions printed like v20.0.0 etc.; this confirms installation.

Run your first Node.js script:


// demo.js
console.log("Hello from Node.js!");


// In terminal:
node demo.js
If the terminal prints Hello from Node.js!, your environment is working.
▸ 1.2.5 Package managers (npm, yarn, pnpm)

JavaScript has a huge ecosystem of reusable code called packages (or modules). Package managers automate installing, updating, and tracking these packages for your project.

▸ npm (Node Package Manager)
  • Comes bundled with Node.js installation.
  • Stores packages in node_modules folder.
  • Uses package.json to describe dependencies and scripts.

// Initialize a new project:
npm init -y

// Install a package (example: lodash):
npm install lodash

// Run a script defined in package.json:
npm run start
In exams, you might be asked what npm init does: it creates a package.json file.
▸ yarn & pnpm (alternatives)
  • yarn: Focused on speed and reliability, earlier used a different lockfile.
  • pnpm: Disk-efficient; shares packages across projects using hard links.

// Example commands (very similar to npm):
yarn add axios
pnpm add axios

For viva: “npm is the default package manager for Node.js; yarn and pnpm are popular alternatives focusing on performance and disk efficiency.”

▸ Real-world workflow example 

// 1) Create a new folder and move into it:
mkdir js-basics
cd js-basics

// 2) Initialize package.json:
npm init -y

// 3) Install a helper library:
npm install dayjs   // small library for dates

// 4) Use it in a file:


// index.js
import dayjs from "dayjs";

console.log("Current time is:", dayjs().format("YYYY-MM-DD HH:mm:ss"));


// 5) Run your script:
node index.js
This pattern (init → install → import → run) repeats in countless real JS projects.
▸ 1.2.6 Version control with Git basics

Git is a distributed version control system. In simple terms, it:

  • Keeps a history of your code changes.
  • Allows you to experiment safely (branches).
  • Makes collaboration easy via platforms like GitHub/GitLab.
▸ Core mental model

Think of Git as a time machine for your project:

  • Each commit is a snapshot of your code with a message.
  • You can jump back and forth between commits.
  • Branches are alternate timelines you can merge later.
▸ Minimum Git commands for JS projects 

// Initialize Git in a folder:
git init

// Check current status:
git status

// Stage files for commit:
git add .

// Commit with a message:
git commit -m "Initial commit"

// Add remote (e.g., GitHub) and push:
git remote add origin <REMOTE_URL>
git push -u origin main
For viva: know that git init creates a repository, git add stages changes, and git commit records them.

▸ 1.3.2 Writing your first JavaScript program
▸ First program in browser console
  1. Open any webpage (even a blank one).
  2. Open DevTools → Console tab.
  3. Type the following and press Enter:

console.log("Hello, world from the browser!");
This is the fastest way to verify that you understand where JavaScript runs and where output appears.
▸ First program in Node.js
  1. Create a file hello.js.
  2. Add this code:

// hello.js
console.log("Hello from Node.js!");
  1. Run it in terminal:

node hello.js

Same console.log, but different environment (terminal instead of browser). This reinforces the idea that JavaScript is a language that can run in multiple environments.

▸ 1.3.3 console.log and debugging basics

console.log() prints information to the Console. It is your first and simplest debugging technique.

▸ Basic usage of console.log 

const a = 10;
const b = 20;
console.log("a =", a);
console.log("b =", b);
console.log("sum =", a + b);
Output will show in browser DevTools Console or in terminal (Node.js).
▸ Debugging a simple bug with console.log 

// Buggy code:
function multiply(x, y) {
    return x + y;  // BUG: should be multiplication
}

console.log(multiply(2, 3)); // Expect 6, get 5

Use console.log to inspect values:


function multiply(x, y) {
    console.log("x =", x, "y =", y);
    const result = x + y;  // still wrong, but we see the flow
    console.log("result =", result);
    return result;
}

You will quickly realize that the operator is wrong, and correct it: 


function multiply(x, y) {
    return x * y;  // fixed
}
▸ 1.3.4 Linking JavaScript to HTML

To create interactive web pages, you must understand how to connect JS files with HTML files.

▸ Inline scripts vs external scripts

Inline script: JS written directly inside HTML.


<!DOCTYPE html>
<html>
<head>
    <title>Inline JS Example</title>
</head>
<body>
    <h1>Hello</h1>

    <script>
        console.log("Running inline JavaScript");
    </script>
</body>
</html>

External script: JS written in a separate file, linked by <script src="...">.


<!-- index.html -->
<!DOCTYPE html>
<html>
<head>
    <title>External JS Example</title>
</head>
<body>
    <h1>Hello</h1>
    <script src="app.js"></script>
</body>
</html>


/* app.js */
console.log("Running external JavaScript file");
External scripts are preferred in real projects because they keep HTML and JS separate, cleaner, and reusable.
▸ 1.3.5 Script tags and loading strategies

How and where you place your <script> tags affects when your JavaScript runs and whether your page feels fast.

▸ Classic pattern: script at the end of body 

<!DOCTYPE html>
<html>
<head>
    <title>Script at Bottom</title>
</head>
<body>
    <h1>My Page</h1>
    <button id="btn">Click me</button>

    <script src="main.js"></script>
</body>
</html>
The browser loads HTML first, then runs JS. So JS can safely access DOM elements.
▸ <script defer> and <script async>

In modern HTML, you often put scripts in <head> with defer or async attributes: 


<head>
    <script src="main.js" defer></script>
</head>
  • defer → Download script in parallel, execute after HTML is parsed.
  • async → Download in parallel, execute as soon as download finishes (may run before HTML is fully parsed).

For most beginner and intermediate projects, use defer. It gives good performance and predictable behavior.

Comments are ignored by the JavaScript engine but read by humans. Good comments explain why, not just what.

// Single-line comment

 Multi-line comment

▸ Example: documenting a function 

/**
 * Calculates the average of an array of numbers.
 * @param {number[]} values - Array of numeric values.
 * @returns {number} The average value, or NaN for empty array.
 */
function average(values) {
    if (values.length === 0) return NaN;
    let sum = 0;
    for (const v of values) {
        sum += v;
    }
    return sum / values.length;
}
This JSDoc style comment can be read by tools (VS Code, TypeScript) to show hints and help.
