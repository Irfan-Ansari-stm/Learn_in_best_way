Module 22: Future of HTML

Explore how HTML is evolving, how standards are created, and how to position yourself for long-term success in the web platform ecosystem.

Executive Summary

Overview

22.1 Upcoming HTML Features

Future HTML features evolve through proposal → experimentation → origin trials → standardization → cross-browser implementation.

22.1.1 Experimental APIs

Experimental HTML & browser APIs often go through “Origin Trials” in Chrome or “Flags” in Firefox/Edge. These may include:

  • File System Access API
  • HTML Modules
  • Cookie Store API
  • Declarative Shadow DOM enhancements
  • Web GPU integration

Example: Declarative Shadow DOM emerging syntax:


<div id="profile-card">
  <template shadowrootmode="open">
    <style>
      .box { padding:1rem; border:1px solid #fff3; }
    </style>
    <div class="box">User Profile</div>
  </template>
  Visible Light DOM fallback
</div>

This enables encapsulation without JS.

22.1.2 Browser implementation timelines

New HTML features follow multi-step adoption:

  1. Proposal stage (discussion in WICG)
  2. Initial implementation in single browser
  3. Origin trial / flag for real-world testing
  4. Standardization in WHATWG spec
  5. Cross-browser adoption

Example timeline model:


2023 Q4 – Proposal written
2024 Q1 – Chromium prototype behind flag
2024 Q3 – WICG discussion & first draft spec
2025 Q1 – Chrome origin trial
2026 – Cross-browser interest (Mozilla, WebKit)
2027 – Standardized / widely adopted

Developers track these via: ChromeStatus, Mozilla Standards Positions, WebKit Feature Status.

22.1.3 Backwards compatibility strategies

When using future HTML features, you must design for older browsers:

  • Progressive enhancement: Start with basic HTML, layer new features on top.
  • Feature detection: Before using an API, check if it exists.
  • Graceful degradation: If a feature is missing, UI still works with reduced power.

Example: feature detection for an experimental API (JS):


if ("cookieStore" in window) {
  // Use modern Cookie Store API
  cookieStore.getAll().then(cookies => {
    console.log("Cookies:", cookies);
  });
} else {
  // Fallback for older browsers
  console.log("Cookie Store API not supported, using document.cookie");
  const legacy = document.cookie.split(";");
  console.log(legacy);
}

Example: using new HTML element with fallback:


<picture>
  <source srcset="photo.avif" type="image/avif">
  <source srcset="photo.webp" type="image/webp">
  <img src="photo.jpg" alt="Portrait">
</picture>

Older browsers will ignore unsupported <source> formats and still render the <img> fallback.

22.2 Web Standards Participation

HTML is not controlled by one company. It evolves through standards bodies and open participation. Understanding these processes helps you anticipate future changes and influence them.

22.2.1 Understanding W3C and WHATWG

WHATWG (Web Hypertext Application Technology Working Group) maintains the living HTML Standard.

W3C (World Wide Web Consortium) produces recommendations and specs (CSS, ARIA, etc.).

Very simplified view:


Browser vendors + experts
        ↓  (discuss proposals)
 WHATWG HTML Living Standard
        ↓
 W3C groups (e.g. WebApps, CSS WG, WAI)
        ↓
 Recommendations, notes, ARIA roles, tests

As a developer, you do not “follow random blogs”; you follow the HTML Living Standard and W3C/WICG discussions to see what is coming.

22.2.2 Contributing to specifications

Yes, even an individual developer can influence future HTML behavior. Typical path:

  1. Identify a problem in real-world projects.
  2. Check if spec already covers it (read WHATWG / W3C docs).
  3. File an issue with clear use-cases and examples.
  4. Participate respectfully in discussion threads.
  5. Optionally join a community group or WICG proposal.

Example: good spec issue structure (pseudocode text):


Title: Proposal: Attribute for shadow-friendly focus management

Summary:
  In large enterprise apps, managing focus across shadow roots is difficult.
  Proposal is to introduce a declarative attribute X that...

Use-cases:
  1) Screen-reader friendly dialogs in shadow DOM.
  2) Complex reusable components spanning multiple roots.

Examples:
  <button autofocus-scope="component">Open dialog</button>

Interoperability / Security:
  - Works with existing focus rules
  - No cross-origin leaks

Even if your idea does not land, the reasoning you see in responses teaches you a lot about platform design.

22.2.3 Community involvement

You don’t have to be a spec author to be part of the web standards ecosystem. Options include:

  • Testing experimental features in nightly browsers and filing bugs.
  • Writing articles that clarify complex platform features.
  • Creating minimal reproducible demos of browser issues.
  • Helping maintain MDN documentation or other docs.

Example: community demo snippet (CodePen / JSFiddle style HTML):


<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>Experimental API Demo</title>
</head>
<body>
  <h1>Cookie Store Demo</h1>
  <button id="read-cookies">Read cookies</button>
  <pre id="output"></pre>

  <script>
    const output = document.getElementById("output");
    document.getElementById("read-cookies").addEventListener("click", async () => {
      if (!("cookieStore" in window)) {
        output.textContent = "Cookie Store API not supported in this browser.";
        return;
      }
      const cookies = await cookieStore.getAll();
      output.textContent = JSON.stringify(cookies, null, 2);
    });
  </script>
</body>
</html>

Such demos help browser vendors understand real-world implications of APIs, and they are extremely valuable.

22.3 Career Development

HTML may look “basic”, but deep expertise in the web platform can power a multi-decade career. This section is about using your HTML knowledge strategically.

22.3.1 Building a professional portfolio

Your portfolio should demonstrate:

  • Semantically structured HTML (articles, sections, asides).
  • Accessible components (ARIA usage, focus management).
  • Performance-conscious markup (lazy images, responsive media).
  • Integration with build tools / SSG / SSR where relevant.

Example: portfolio project index snippet:


<main>
  <section aria-labelledby="proj-heading">
    <h2 id="proj-heading">Selected Projects</h2>
    <article class="project-card">
      <header>
        <h3>Accessible Design System</h3>
        <p>HTML, CSS, Web Components</p>
      </header>
      <section>
        <p>Built reusable accessible components for a large design system.</p>
      </section>
      <footer>
        <a href="https://example.com/demo">Live demo</a>
        <a href="https://github.com/user/repo">Source</a>
      </footer>
    </article>
  </section>
</main>

This communicates seriousness about semantics and accessibility to hiring teams.

22.3.2 Contributing to open source

Open source is one of the strongest accelerators for your HTML/web career:

  • Fixing small documentation issues in popular frameworks.
  • Improving HTML semantics in component libraries.
  • Creating accessible starter templates / boilerplates.
  • Writing sample apps that showcase best practices.

Example: improving a library’s HTML template (before/after):

Before:


<div class="card">
  <div>Title</div>
  <div>Body...</div>
</div>

After (more semantic, accessible):


<article class="card" aria-labelledby="card-title">
  <header>
    <h2 id="card-title">Title</h2>
  </header>
  <section>
    <p>Body...</p>
  </section>
</article>

Small changes like this demonstrate deep understanding of HTML’s purpose.

22.3.3 Staying current with web standards

To stay up-to-date:

  • Follow official sources: WHATWG HTML spec, W3C news, MDN changelogs.
  • Track browser release notes (Chrome, Firefox, Safari, Edge).
  • Subscribe to web platform newsletters / podcasts.
  • Regularly revisit production sites you built and re-measure performance & accessibility.

Personal “maintenance” pseudocode for your skills:


Every month:
  - Read spec change summaries
  - Check MDN for new HTML / DOM features
  - Audit 1 of your old projects
  - Update your portfolio with a small improvement

Every 6–12 months:
  - Learn a new testing tool or devtool
  - Try a new build/deployment variation (SSG, SSR, etc.)
  - Present a small talk or write an article

This rhythm ensures you keep pace with HTML’s evolution without burning out.
