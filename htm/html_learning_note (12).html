Module 7 · Tables

HTML Tables: Structure, Power, and Accessibility

Tables are essential for representing structured data: schedules, mark sheets, pricing plans, research measurements, financial reports, etc. In this module, you will learn not only the basic <table> syntax, but also how to structure complex tables, keep them accessible, and handle responsiveness.

Executive Overview of Module 7
High-level map before details
  • 7.1 Basic Table Structure: Understand how <table>, <tr>, <td>, and <th> work together, how header cells are defined, and how <caption> describes the table.
  • 7.2 Advanced Table Features: Learn to logically organize large tables into <thead>, <tbody>, <tfoot>, style columns using <colgroup> and <col>, and span cells with colspan and rowspan.
  • 7.3 Table Accessibility: Discover how to correctly associate headers and data cells, provide summary/description, create responsive tables without breaking semantics, and avoid misusing tables for layout.
Big idea: A table is not just a grid of boxes. It is a structured representation of relationships between headers and data cells. Good HTML makes those relationships explicit so both humans and assistive technologies can understand the data.

7.1 Basic Table Structure From grid to semantic data

At a basic level, an HTML table is a set of rows and columns. But behind that simple grid, there is clear structure: the table itself, rows, header cells, and data cells. Once you understand these building blocks, more advanced features become easy.

7.1.1 <table>, <tr>, <td>, <th> elements
Core table skeleton

The core elements are:

  • <table> — container for the whole table.
  • <tr> — table row.
  • <td> — table data cell (normal cell).
  • <th> — table header cell (usually bold and centered by default).

Example: simple schedule table.

<table>
    <tr>
        <th>Day</th>
        <th>Time</th>
        <th>Subject</th>
    </tr>
    <tr>
        <td>Monday</td>
        <td>9:00 - 10:00</td>
        <td>HTML Fundamentals</td>
    </tr>
    <tr>
        <td>Wednesday</td>
        <td>11:00 - 12:00</td>
        <td>CSS Basics</td>
    </tr>
</table>
Visually this already looks like a table, but we will soon improve it with <thead>/<tbody>, <caption>, and accessibility attributes.
7.1.2 Table headers and scope
<th> and the scope attribute

Header cells (<th>) describe the rows or columns of data. For assistive technologies, it is very useful to specify the scope of each header:

  • scope="col" — header for a column.
  • scope="row" — header for a row.
  • scope="colgroup" — header for a group of columns.
  • scope="rowgroup" — header for a group of rows.

Example: mark sheet with row and column headers.

<table>
    <tr>
        <th scope="col">Student</th>
        <th scope="col">HTML</th>
        <th scope="col">CSS</th>
        <th scope="col">JavaScript</th>
    </tr>
    <tr>
        <th scope="row">Irfan</th>
        <td>90</td>
        <td>88</td>
        <td>85</td>
    </tr>
    <tr>
        <th scope="row">Sara</th>
        <td>92</td>
        <td>91</td>
        <td>89</td>
    </tr>
</table>
For a sighted user, headers are obvious visually. For a screen reader user, scope helps the device announce: “HTML: 90” as “Irfan, HTML: 90” instead of just “90”.
7.1.3 Caption element (<caption>)
Naming the whole table

The <caption> element gives the table a title or description. It is placed immediately after the opening <table> tag. 

<table>
    <caption>
        Table 1: Exam scores of students in front-end subjects.
    </caption>
    <tr>
        <th scope="col">Student</th>
        <th scope="col">HTML</th>
        <th scope="col">CSS</th>
        <th scope="col">JavaScript</th>
    </tr>
    <tr>
        <th scope="row">Irfan</th>
        <td>90</td>
        <td>88</td>
        <td>85</td>
    </tr>
    <tr>
        <th scope="row">Sara</th>
        <td>92</td>
        <td>91</td>
        <td>89</td>
    </tr>
</table>
<caption> improves accessibility and also acts as a visible label. It is more semantically correct than writing “Table 1” in a separate <p> tag.

7.2 Advanced Table Features Structure for complex data

Real-world tables can be long, wide, and complex. Grouping rows into header/body/footer sections, styling columns, and spanning cells across rows or columns helps you create readable, maintainable tables.

7.2.1 Table sections (<thead>, <tbody>, <tfoot>)
Logical grouping of rows

Table sections help you separate header rows, main data rows, and footer (summary) rows.

  • <thead> — usually contains column headers.
  • <tbody> — main body of data rows.
  • <tfoot> — summary, totals, or notes.

Example: sales data with total row in <tfoot>.

<table>
    <caption>Table 2: Quarterly sales by region.</caption>

    <thead>
        <tr>
            <th scope="col">Region</th>
            <th scope="col">Q1</th>
            <th scope="col">Q2</th>
            <th scope="col">Q3</th>
            <th scope="col">Q4</th>
        </tr>
    </thead>

    <tbody>
        <tr>
            <th scope="row">North</th>
            <td>120</td>
            <td>135</td>
            <td>140</td>
            <td>150</td>
        </tr>
        <tr>
            <th scope="row">South</th>
            <td>110</td>
            <td>115</td>
            <td>130</td>
            <td>140</td>
        </tr>
    </tbody>

    <tfoot>
        <tr>
            <th scope="row">Total</th>
            <td>230</td>
            <td>250</td>
            <td>270</td>
            <td>290</td>
        </tr>
    </tfoot>
</table>
Browsers and assistive technologies can use these sections to provide better navigation, repeat headers when printing, and style each section differently using CSS.
7.2.2 Column groups (<colgroup>, <col>)
Styling entire columns

Columns are not explicit elements like rows and cells, but you can style them using <colgroup> and <col>. 

<table>
    <caption>Table 3: Student marks with highlighted subject columns.</caption>

    <colgroup>
        <col>               <!-- Student column -->
        <col class="col-html">
        <col class="col-css">
        <col class="col-js">
    </colgroup>

    <thead>
        <tr>
            <th scope="col">Student</th>
            <th scope="col">HTML</th>
            <th scope="col">CSS</th>
            <th scope="col">JavaScript</th>
        </tr>
    </thead>

    <tbody>
        <tr>
            <th scope="row">Irfan</th>
            <td>90</td>
            <td>88</td>
            <td>85</td>
        </tr>
        <tr>
            <th scope="row">Sara</th>
            <td>92</td>
            <td>91</td>
            <td>89</td>
        </tr>
    </tbody>
</table>

Conceptual CSS for styling columns:

/* Example column styling */
.col-html {
    background-color: rgba(6, 182, 212, 0.1);
}

.col-css {
    background-color: rgba(245, 158, 11, 0.1);
}

.col-js {
    background-color: rgba(34, 197, 94, 0.1);
}
<colgroup>/<col> are very useful when you need to apply styles to an entire column (borders, background, width) without manually adding classes to every <td> or <th> in that column.
7.2.3 Spanning cells (colspan, rowspan)
Merging rows and columns

colspan and rowspan allow a cell to span multiple columns or rows, respectively.

  • colspan="2" — cell takes width of two columns.
  • rowspan="3" — cell takes height of three rows.

Example: timetable with a merged “Lab” session spanning two hours.

<table>
    <caption>Table 4: Weekly timetable with merged cells.</caption>

    <thead>
        <tr>
            <th scope="col">Day</th>
            <th scope="col">9:00 - 10:00</th>
            <th scope="col">10:00 - 11:00</th>
        </tr>
    </thead>

    <tbody>
        <tr>
            <th scope="row">Monday</th>
            <td>HTML Theory</td>
            <td>CSS Basics</td>
        </tr>
        <tr>
            <th scope="row">Wednesday</th>
            <td colspan="2">Front-end Lab</td>
        </tr>
    </tbody>
</table>

Example: using rowspan for grouping rows by category.

<table>
    <caption>Table 5: Course modules grouped by category.</caption>

    <thead>
        <tr>
            <th scope="col">Category</th>
            <th scope="col">Module</th>
            <th scope="col">Topics</th>
        </tr>
    </thead>

    <tbody>
        <tr>
            <th scope="row" rowspan="2">HTML</th>
            <td>Module 6</td>
            <td>Images and Media</td>
        </tr>
        <tr>
            <td>Module 7</td>
            <td>Tables</td>
        </tr>
        <tr>
            <th scope="row" rowspan="2">CSS</th>
            <td>Module 8</td>
            <td>Selectors</td>
        </tr>
        <tr>
            <td>Module 9</td>
            <td>Layout</td>
        </tr>
    </tbody>
</table>
Be careful with colspan/rowspan: the total number of cells per row must still form a valid grid. If you misuse these attributes, the table structure becomes confusing for both browsers and assistive technologies.

7.3 Table Accessibility Readable for everyone

Data tables can easily become unusable for people using screen readers or small devices. In this section you will learn how to explicitly connect headers to cells, describe tables clearly, and handle responsiveness without breaking semantics.

7.3.1 Proper header association
scope vs headers/id

For simple tables, using scope on <th> elements is enough. For more complex tables (multiple header rows/columns), you may need the headers and id attributes.

Simple association using scope (already shown earlier):

<th scope="col">HTML</th>
<th scope="row">Irfan</th>

Complex association using headers:

<table>
    <caption>Table 6: Exam scores with midterm and final breakdown.</caption>

    <thead>
        <tr>
            <th id="h-student">Student</th>
            <th id="h-html-mid">HTML Mid</th>
            <th id="h-html-final">HTML Final</th>
            <th id="h-css-mid">CSS Mid</th>
            <th id="h-css-final">CSS Final</th>
        </tr>
    </thead>

    <tbody>
        <tr>
            <th id="s-irfan">Irfan</th>
            <td headers="h-html-mid s-irfan">45</td>
            <td headers="h-html-final s-irfan">48</td>
            <td headers="h-css-mid s-irfan">44</td>
            <td headers="h-css-final s-irfan">46</td>
        </tr>
    </tbody>
</table>
Screen readers can now announce each cell with its corresponding headers, e.g.: “Irfan, HTML Mid: 45”. This is vital in complex multi-dimensional tables.
7.3.2 Summary and description
Explaining what the table shows

Historically, there was a summary attribute on <table>, but it has been deprecated in HTML5. Modern practice is:

  • Use <caption> for a short title.
  • Use nearby text (paragraphs) to describe patterns, trends, or how to read the table.
  • Optionally use ARIA attributes such as aria-describedby to associate longer descriptions.

Example: caption + descriptive paragraph.

<p id="sales-desc">
    The table below shows quarterly sales for each region. Values are in thousands of units.
</p>

<table aria-describedby="sales-desc">
    <caption>Table 7: Quarterly sales by region (in thousands).</caption>
    <thead>
        <tr>
            <th scope="col">Region</th>
            <th scope="col">Q1</th>
            <th scope="col">Q2</th>
            <th scope="col">Q3</th>
            <th scope="col">Q4</th>
        </tr>
    </thead>
    <tbody>
        <tr>
            <th scope="row">North</th>
            <td>120</td>
            <td>135</td>
            <td>140</td>
            <td>150</td>
        </tr>
        <tr>
            <th scope="row">South</th>
            <td>110</td>
            <td>115</td>
            <td>130</td>
            <td>140</td>
        </tr>
    </tbody>
</table>
This pattern helps all users: a caption to quickly identify the table, and a description to explain context and how to interpret the numbers.
7.3.3 Responsive table techniques
Horizontal scroll vs stacking

Tables are naturally wide. Making them responsive is tricky because you do not want to break the relationships between rows and columns. Common patterns:

  • Horizontal scrolling: Keep table semantics; allow overflow on small screens.
  • Stacked layout: Convert each row into a card-like structure with labels.
  • Column hiding: Hide less important columns on small screens (careful: may hide important data).

Example: simple scrollable table container (semantic-friendly).

<div class="table-responsive">
    <table>
        <caption>Table 8: Simple responsive table using horizontal scroll.</caption>
        <thead>
            <tr>
                <th scope="col">Student</th>
                <th scope="col">HTML</th>
                <th scope="col">CSS</th>
                <th scope="col">JavaScript</th>
                <th scope="col">React</th>
            </tr>
        </thead>
        <tbody>
            <tr>
                <th scope="row">Irfan</th>
                <td>90</td>
                <td>88</td>
                <td>85</td>
                <td>80</td>
            </tr>
            <tr>
                <th scope="row">Sara</th>
                <td>92</td>
                <td>91</td>
                <td>89</td>
                <td>84</td>
            </tr>
        </tbody>
    </table>
</div>

Conceptual CSS (Bootstrap already provides .table-responsive):

.table-responsive {
    overflow-x: auto;
}
Horizontal scrolling is often the least risky pattern because it preserves table semantics. Stacked layouts and column hiding can be useful but must be designed carefully so users do not lose important information.
7.3.4 Data table vs layout table considerations
Modern best practice

Historically, developers used tables for page layout (multiple columns, header/footer, etc.). In modern HTML, this is considered bad practice.

  • Data tables: tables that represent relationships between data points. Use <table> + semantic markup (<th>, <caption>, etc.).
  • Layout tables: tables used purely to arrange visual elements. Should be avoided; use CSS layout (flexbox, grid) instead.

Bad (layout) usage example — do not do this in modern code:

<!-- Bad practice: using table for layout -->
<table>
    <tr>
        <td>Sidebar content</td>
        <td>Main content</td>
    </tr>
</table>

Modern alternative using CSS:

<div class="layout">
    <aside>Sidebar content</aside>
    <main>Main content</main>
</div>
.layout {
    display: grid;
    grid-template-columns: 250px 1fr;
    gap: 1rem;
}
Assistive technologies treat tables as structured data. If you misuse tables for layout, screen readers may announce “table with X rows and Y columns” where the user expects a simple content layout. This is confusing and harms accessibility.
Practice ideas:
  • Create a mark sheet table with <caption>, <thead>, <tbody>, <tfoot>, and proper scopes.
  • Build a timetable using colspan and ensure the grid still makes sense.
  • Make a table responsive using a scrollable container, test on a mobile-sized viewport.
  • Explain to someone (or yourself in notes) the difference between data tables and layout tables, with at least one example of each (and why layout tables are discouraged).
