Module 6 · Images and Media

Making the Web Visual, Fast, and Accessible

In this module you will go beyond “just show an image” and learn how to handle images and media in a way that works well for real websites: fast loading, visually sharp on different devices, and accessible to assistive technologies like screen readers.

Focus: Practical HTML for production-ready sites <img> basics alt text responsive srcset/sizes <picture> art direction semantic figures audio and video accessibility and performance
Executive Overview of Module 6
Quick mental map before details
  • 6.1 Basic Images: Learn the core <img> syntax, the critical role of alt, and why width/height attributes matter for layout stability and performance.
  • 6.2 Responsive Images: Understand how srcset, sizes, and <picture> allow you to serve different files to different devices (art direction + modern formats like WebP).
  • 6.3 Figure and Caption: Use <figure> and <figcaption> to give images semantic meaning, not just visual layout, and build clean image galleries.
  • 6.4 Audio and Video: Use native HTML5 <audio> and <video> with multiple formats, poster images, captions and transcripts, and accessibility considerations.
Big idea: Images and media are often the heaviest part of a page. Good developers think about what they show (content), how they show it (HTML/CSS), and how efficiently they deliver it (formats, responsive techniques, accessibility).

6.1 Basic Images The <img> building block

The <img> element is the core way to display images in HTML. In this section you will learn how to use it correctly, how alt text affects accessibility and SEO, and how to avoid layout jumps using width and height.

6.1.1 <img> element and essential attributes
src and alt in practice

The <img> element is a void element: it has no closing tag. At minimum you should provide a src and an alt. 

<img src="profile.jpg" alt="Profile photo of the author">
  • src — URL of the image file (relative or absolute).
  • alt — alternative text describing the image for users who cannot see it (screen readers, broken images, text-only clients).

A simple profile card that combines <img> with text content:

<div class="profile-card">
    <img src="images/irfan-profile.jpg"
         alt="Irfan smiling and looking at the camera"
         width="200" height="200">

    <h2>Irfan</h2>
    <p>Frontend Developer and Educator</p>
</div>
Notice that we already set width and height. This allows the browser to reserve the correct space even before the image finishes loading, which reduces layout shifts.
6.1.2 Alt text for accessibility
Screen readers, SEO, broken images

The alt attribute is not just a “nice to have”. It is central for accessibility and also helps search engines understand the image. 

<img src="chart-sales.png"
     alt="Bar chart showing monthly sales from January to June, peaking in May">
  • Informative images: Describe what the image conveys. Imagine someone on a screen reader; what would you say in one short sentence?
  • Decorative images: Use empty alt: alt="". This tells assistive technology to skip it.
  • Functional images: For icons inside buttons/links, describe the function, not the icon shape (e.g. “Search” instead of “Magnifying glass icon”). 
<!-- Decorative flourish, should be ignored by screen readers -->
<img src="shapes/bg-wave.svg" alt="" aria-hidden="true">

<!-- Functional icon inside a button -->
<button type="submit">
    <img src="icons/search.svg" alt="Search">
</button>
Good practice Exam/interview topic Many accessibility interviews include a question like: “Explain the purpose of alt attributes and when you might leave them empty.”
6.1.3 Width and height attributes
Aspect ratio and layout stability

The width and height attributes define the intrinsic dimensions of the image. From these, the browser can compute the aspect ratio. 

<img src="hero.jpg"
     alt="Landing page hero"
     width="1200" height="600">
  • This image has a 2:1 aspect ratio (1200 ÷ 600).
  • The browser reserves space with that ratio before the image loads, reducing layout shifts.
  • You can still make the image responsive with CSS by constraining its max width and letting height adjust automatically.

Common responsive pattern (often with Bootstrap's img-fluid):

<img src="blog-cover.jpg"
     alt="Students studying with laptops"
     width="1600" height="900"
     class="img-fluid">
/* Conceptual CSS: Bootstrap's img-fluid does something similar */
img {
    max-width: 100%;
    height: auto;
}
A typical interview answer: “We should set width/height for images to preserve aspect ratio and avoid layout shifts, which improves Core Web Vitals like CLS (Cumulative Layout Shift).”
6.1.4 Image formats and optimization
JPEG, PNG, SVG, WebP

Different image formats are optimized for different use cases:

  • JPEG / JPG: Photographs and complex scenes. Lossy compression, small file size.
  • PNG: Images needing transparency or crisp edges (logos, UI icons). Lossless, usually larger than JPEG.
  • SVG: Vector graphics (logos, icons, simple illustrations). Infinitely scalable without losing quality.
  • WebP / AVIF: Modern formats with better compression; smaller files at similar quality, but require fallback for older browsers. 
<!-- SVG logo: sharp at any size -->
<img src="logo.svg" alt="Brand logo" width="120" height="40">

<!-- PNG icon with transparency -->
<img src="icons/check.png" alt="" width="24" height="24">
Performance Images usually dominate the total page weight. Choosing the right format and compressing images properly can dramatically reduce load time, especially on mobile networks.

6.2 Responsive Images Serve the right file to each device

Responsive images are about more than “shrinking with CSS”. You want to send smaller files to small/low-density screens and sharper files to high-density displays, without wasting bandwidth.

6.2.1 srcset for different resolutions
1x vs 2x vs 3x density

The srcset attribute allows you to provide multiple versions of the same image at different resolutions. The browser picks the best one based on the device’s pixel density. 

<img
    src="avatar-1x.jpg"
    srcset="avatar-1x.jpg 1x, avatar-2x.jpg 2x, avatar-3x.jpg 3x"
    alt="User avatar"
    width="80" height="80">
  • 1x: standard-density screens.
  • 2x: Retina / high-density screens (many smartphones).
  • 3x: very high-density displays.
Without srcset, you may end up always serving a heavy, high-resolution image even to low-end devices on slow networks.
6.2.2 sizes attribute for responsive layouts
Tell the browser the expected layout width

When you use srcset with width descriptors (like 400w, 800w), you should also provide a sizes attribute. It tells the browser how wide the image will appear in different viewport conditions. 

<img
    src="hero-800.jpg"
    srcset="
        hero-400.jpg 400w,
        hero-800.jpg 800w,
        hero-1200.jpg 1200w,
        hero-1600.jpg 1600w
    "
    sizes="
        (max-width: 600px) 100vw,
        (max-width: 1024px) 80vw,
        1200px
    "
    alt="Campus hero image"
    width="1600" height="900">
  • 400w, 800w, … describe the intrinsic widths of each file.
  • (max-width: 600px) 100vw means: if the viewport is at most 600px wide, the image will take approximately 100% of the viewport width.
  • The final 1200px is the default assumed display width for larger viewports.
Conceptually, srcset + sizes says: “Here are my candidate files and here is how wide the image will appear in each situation. Browser, please choose the most appropriate file.”
6.2.3 <picture> element for art direction
Different crops for mobile vs desktop

Sometimes you need different crops or completely different images for different devices (for example, a tight vertical crop on mobile vs a wide hero banner on desktop). This is called art direction, and the <picture> element supports it. 

<picture>
    <!-- Mobile: vertical crop -->
    <source
        srcset="hero-portrait.webp"
        type="image/webp"
        media="(max-width: 600px)">

    <!-- Desktop: wide crop -->
    <source
        srcset="hero-wide.webp"
        type="image/webp"
        media="(min-width: 601px)">

    <!-- Fallback for older browsers -->
    <img
        src="hero-wide.jpg"
        alt="Students collaborating in a classroom"
        width="1600" height="900">
</picture>
  • <source> elements define conditional sources with media queries and formats.
  • The final <img> is a required fallback for browsers that do not support <picture> or a given format.
6.2.4 WebP and modern image formats
Better compression with graceful fallback

Modern formats such as WebP and AVIF provide significantly better compression than JPEG or PNG for many images. The <picture> element lets you offer these formats with safe fallback. 

<picture>
    <source srcset="banner.avif" type="image/avif">
    <source srcset="banner.webp" type="image/webp">
    <img src="banner.jpg" alt="Course banner" width="1200" height="500">
</picture>
Browsers will use AVIF if supported, otherwise fall back to WebP, and then to JPEG. This sequence gives you modern performance without sacrificing compatibility.

6.3 Figure and Caption Giving images semantic meaning

The <figure> and <figcaption> elements give structure and meaning to images, diagrams, code listings, or even videos by grouping them with their caption in a semantic way.

6.3.1 <figure> and <figcaption> elements
Images with explicit captions

Basic structure for a figure: 

<figure>
    <img src="lab-setup.jpg"
         alt="Physics lab setup with laser and mirrors"
         width="800" height="450">
    <figcaption>
        Figure 1: Experimental setup used for measuring the refractive index of glass.
    </figcaption>
</figure>
  • <figure> represents a self-contained piece of content (image, chart, code snippet, etc.) that is referenced from the main flow.
  • <figcaption> provides the caption or explanation associated with that figure.
  • Screen readers identify this as a figure with a caption, which improves navigation for visually impaired users.
6.3.2 Semantic image grouping vs layout-only grouping
Why <figure> matters

Compare a purely visual grouping: 

<!-- Layout-only grouping using a generic div -->
<div class="image-block">
    <img src="earth.png" alt="Earth">
    <p>Earth viewed from space.</p>
</div>

with a semantic figure: 

<figure>
    <img src="earth.png" alt="Earth viewed from space">
    <figcaption>
        Figure 2: Earth observed from the Apollo 17 mission in 1972.
    </figcaption>
</figure>
Benefits of using <figure>:
  • Assistive technologies can announce “Figure” and read the caption clearly.
  • Search engines can treat the figure and caption as a meaningful unit.
  • In long documents, you can build a “List of Figures” automatically based on figures.
6.3.3 Image galleries (semantic + responsive)
Combining <figure> with Bootstrap grid

A simple responsive gallery using Bootstrap classes and semantic figures: 

<div class="row g-3">
    <figure class="col-6 col-md-4">
        <img src="gallery/campus-1.jpg"
             alt="Main campus building"
             class="img-fluid">
        <figcaption>Main campus building</figcaption>
    </figure>

    <figure class="col-6 col-md-4">
        <img src="gallery/campus-2.jpg"
             alt="Library interior"
             class="img-fluid">
        <figcaption>Central library</figcaption>
    </figure>

    <figure class="col-6 col-md-4">
        <img src="gallery/campus-3.jpg"
             alt="Computer lab"
             class="img-fluid">
        <figcaption>Computer lab</figcaption>
    </figure>
</div>
In production, you might add a lightbox (click to enlarge), keyboard shortcuts, and ARIA roles. But even without JavaScript, this HTML structure is clean and accessible.

6.4 Audio and Video Elements Native HTML5 media

HTML5 introduced native <audio> and <video> elements, so you do not need plugins like Flash. These elements support multiple formats, built-in controls, captions, and accessibility features.

6.4.1 <audio> element and controls
Play, pause, multiple formats

A basic audio player with two common formats: 

<audio controls>
    <source src="audio/lecture-intro.ogg" type="audio/ogg">
    <source src="audio/lecture-intro.mp3" type="audio/mpeg">
    Your browser does not support the audio element.
</audio>
  • controls shows the built-in play/pause UI.
  • Inside <audio>, multiple <source> elements allow you to support different formats.
  • The text fallback is for very old browsers; in modern browsers it is rarely seen but still good practice.
6.4.2 <video> element and attributes
controls, autoplay, loop, muted

A minimal video player looks like this: 

<video controls width="640" height="360">
    <source src="video/demo.webm" type="video/webm">
    <source src="video/demo.mp4" type="video/mp4">
    Your browser does not support the video tag.
</video>
  • controls — show basic UI (play/pause, volume, timeline).
  • autoplay — attempt to start playing automatically (but modern browsers often block autoplay with sound).
  • loop — restart from the beginning after reaching the end.
  • muted — start without sound; often required for autoplay to work.
  • playsinline — on mobile, play inside the page instead of switching to fullscreen (especially iOS).
<video
    controls
    autoplay
    muted
    loop
    playsinline
    width="800" height="450">

    <source src="promo.webm" type="video/webm">
    <source src="promo.mp4" type="video/mp4">

    Your browser does not support HTML5 video.
</video>
6.4.3 Multiple source formats for media
Cross-browser compatibility

Not all browsers support all audio/video formats, so it is common to provide at least two.

  • Audio: usually audio/ogg and audio/mpeg (MP3).
  • Video: usually video/webm and video/mp4.
<video controls>
    <source src="lecture-part1.webm" type="video/webm">
    <source src="lecture-part1.mp4" type="video/mp4">
    Your browser does not support HTML5 video.
</video>
The browser checks the <source> elements from top to bottom and picks the first one it can play.
6.4.4 Poster images for videos
Nice first frame / thumbnail

The poster attribute defines an image that is shown before the video starts playing (or if it cannot load the first frame quickly). 

<video
    controls
    width="800" height="450"
    poster="thumbnails/lesson-1.png">

    <source src="lesson-1.webm" type="video/webm">
    <source src="lesson-1.mp4" type="video/mp4">
    Your browser does not support HTML5 video.
</video>
A good poster image improves perceived quality. Instead of a blank black rectangle, users see a meaningful thumbnail that matches your branding and content.
6.4.5 Accessibility considerations for media
Captions, transcripts, keyboard

Accessible media generally includes:

  • Captions: synchronized text of spoken content for deaf/hard-of-hearing users.
  • Transcripts: full text versions of audio/video content; searchable and screen-reader friendly.
  • Keyboard accessibility: all controls should be usable via keyboard (default browser controls usually are).

Basic example of captions using WebVTT:

<video controls>
    <source src="talk.mp4" type="video/mp4">
    <track
        kind="captions"
        src="talk-en.vtt"
        srclang="en"
        label="English">
</video>

A simple pattern for including a transcript in HTML: 

<section aria-labelledby="talk1-title">
    <h2 id="talk1-title">Talk 1: Introduction to HTML Media</h2>

    <video controls>
        <source src="talk1.mp4" type="video/mp4">
        Your browser does not support HTML5 video.
    </video>

    <h3>Transcript</h3>
    <p>[00:00] Welcome to this session on HTML media...</p>
    <p>[00:30] First, let's discuss the &lt;audio&gt; element...</p>
    <!-- additional transcript content -->
</section>
In many countries, captions and transcripts are not just “nice UX” but part of legal accessibility requirements for educational or governmental content.
Next steps for practice:
  • Take an existing page and add proper alt text to every image.
  • Convert at least one large hero image to use srcset and sizes.
  • Wrap one important image or diagram in <figure> and <figcaption> and observe how it feels semantically cleaner.
  • Create a simple page with one <audio> and one <video> element, both with appropriate sources and at least one accessibility enhancement (captions or a transcript).
