In this module we go beyond human-readable HTML and move into machine-readable annotations. 0. Executive Overview

Why structured data matters Big Picture

From a search engine’s perspective, most HTML is just text with a rough layout. Structured data adds a precise, machine-readable layer: instead of “some bold text”, you say “this is a Person with name, affiliation, and email”. Microdata is one way to embed such semantics directly into existing HTML using attributes like itemscope, itemtype, and itemprop.

Executive summary of Module 12 TL;DR
  • 12.1 Introduction: You learn what structured data is, how Schema.org defines a shared vocabulary, and how microdata attributes (itemscope, itemtype, itemprop) map that vocabulary into HTML.
  • 12.2 Schema Types: You see concrete patterns for describing people, organizations, products, reviews, events, places, and articles using microdata and JSON-LD, with attention to research-grade correctness.
  • 12.3 Validation & SEO: You learn how to use Google’s Rich Results Test and other validators to verify your markup, debug errors, and understand the SEO impact (rich snippets, knowledge panels, and improved CTR).

Conceptually, you can think of microdata as attaching “typed edges” to your HTML nodes, turning your web page into a small knowledge graph fragment that can be merged into bigger graphs like those of search engines or digital assistants.

12.1

Introduction to Microdata Conceptual Foundations

Microdata is a W3C specification that lets you annotate your HTML with semantic information using attributes. It is one of several ways (alongside RDFa and JSON-LD) to express structured data for search engines and other consumers.

Unstructured: Plain text and HTML — humans understand, machines guess.
Semi-structured: HTML with headings and lists — some hints, but still ambiguous.
Structured: Explicit typed properties (Person, Event, Product) — machine interpretable.
What is structured data? High-level
  • Structured data is information organized according to a fixed schema: there are named fields with well-defined meanings (e.g., name, startDate, price).
  • In the context of the web, structured data usually means markup that identifies entities and their properties in a way that search engines and other agents can reliably parse.
  • Historically, HTML tags were designed for presentation and basic semantics. Structured data adds a domain-specific layer on top: now your “box of text” is explicitly a Product with a reviewRating, and your “date string” is an Event startDate.

You can think of structured data as writing a mini ontology instance for every relevant object on the page. At research level, this connects directly with knowledge graphs, ontologies, and semantic web standards.

12.1.2 Schema.org vocabulary

Schema.org as a shared dictionary Concept
  • Schema.org is a collaborative, community-driven vocabulary for structured data on the web, initiated by major search engines.
  • It defines types (e.g., Person, Organization, Product, Event, Article) and properties (e.g., name, jobTitle, startDate, offers).
  • Microdata uses Schema.org URLs via the itemtype attribute, e.g. itemtype="https://schema.org/Person" .

Conceptually, Schema.org = vocabulary + relationships; microdata = embedding that vocabulary into HTML instances.

Example: simple Person microdata Example 

<div itemscope itemtype="https://schema.org/Person">
  <span itemprop="name">Dr. A. Researcher</span>
  <span itemprop="jobTitle">Assistant Professor</span>
  <span itemprop="affiliation">
    National Institute of Technology
  </span>
  <a href="mailto:a.researcher@example.edu"
     itemprop="email">
    a.researcher@example.edu
  </a>
</div>

Here, the <div> is a Person; nested elements supply properties like name and email.

12.1.3 itemscope & itemtype attributes

Microdata core attributes Syntax
  • itemscope: Declares that the element and its descendants form a single item (entity).
  • itemtype: Provides one or more URLs that identify the type of the item, typically a Schema.org URL.
  • itemprop: Indicates that the element’s content contributes a value for a named property of the enclosing item.
Example: itemscope, itemtype, itemprop together Example 

<article itemscope itemtype="https://schema.org/ScholarlyArticle">
  <h1 itemprop="headline">
    Microdata for Scientific Publishing
  </h1>

  <p>By
    <span itemprop="author" itemscope itemtype="https://schema.org/Person">
      <span itemprop="name">Dr. A. Researcher</span>
    </span>
  </p>

  <time itemprop="datePublished" datetime="2025-11-24">
    24 November 2025
  </time>

  <div itemprop="publisher" itemscope itemtype="https://schema.org/Organization">
    <span itemprop="name">Journal of Web Semantics</span>
  </div>
</article>

Notice how multiple items nest: the article itself (ScholarlyArticle), an embedded Person as author, and an Organization as publisher.

12.2

Common Schema Types Practical Patterns

Rather than memorizing the entire Schema.org universe, we focus on high-impact types that appear frequently in real-world systems: persons, organizations, products, reviews, events, places, and articles/blog posts.

12.2.1 Person & Organization schemas

Modeling people & organizations Concept + Syntax

Person and Organization sit at the core of many knowledge graphs: authors, institutions, companies, research groups, labs, etc.

  • Typical Person properties: name, jobTitle, affiliation, email, url.
  • Typical Organization properties: name, url, logo, address, sameAs (links to profiles, e.g. Wikipedia, LinkedIn).
Example: Person + Organization microdata Example 

<div itemscope itemtype="https://schema.org/Person">
  <span itemprop="name">Irfan Khan</span>
  <span itemprop="jobTitle">Senior Web Engineer</span>

  <div itemprop="affiliation" itemscope itemtype="https://schema.org/Organization">
    <span itemprop="name">DeepWeb Labs</span>
    <a itemprop="url" href="https://deepweblabs.example">
      deepweblabs.example
    </a>
  </div>

  <a href="mailto:irfan@example.com" itemprop="email">
    irfan@example.com
  </a>
</div>

12.2.2 Product & Review schemas

Why Product & Review matter Concept

For e-commerce or SaaS, structured data for Product and Review unlocks rich snippets: price, rating stars, availability, etc.

  • Product properties: name, image, description, brand, offers.
  • Review properties: reviewRating, author, reviewBody, itemReviewed.
Example: Product with aggregateRating (microdata) Example 

<div itemscope itemtype="https://schema.org/Product">
  <span itemprop="name">Noise-Cancelling Headphones X200</span>

  <img src="/img/x200.png"
       alt="Headphones X200"
       itemprop="image" />

  <p itemprop="description">
    Over-ear wireless headphones with 35 dB active noise cancellation.
  </p>

  <div itemprop="brand" itemscope itemtype="https://schema.org/Organization">
    <span itemprop="name">AudioSphere</span>
  </div>

  <div itemprop="offers" itemscope itemtype="https://schema.org/Offer">
    Price:
    <span itemprop="priceCurrency">INR</span>
    <span itemprop="price">8999</span>
    <link itemprop="availability"
          href="https://schema.org/InStock" />
  </div>

  <div itemprop="aggregateRating" itemscope itemtype="https://schema.org/AggregateRating">
    Rated <span itemprop="ratingValue">4.6</span>/5
    based on <span itemprop="reviewCount">128</span> reviews.
  </div>
</div>

12.2.3 Event & Place schemas

Modeling events and locations Concept

For conferences, workshops, concerts, or meetups, Event and Place types help search engines show event cards with dates, locations, and ticket links.

  • Event properties: name, startDate, endDate, location, organizer, eventStatus.
  • Place properties: name, address, geo (lat/long), etc.
Example: Event with Place microdata Example 

<div itemscope itemtype="https://schema.org/Event">
  <h2 itemprop="name">
    International Conference on Semantic Web 2026
  </h2>

  <time itemprop="startDate" datetime="2026-02-15T09:00:00+05:30">
    15 February 2026, 9:00 AM IST
  </time>

  <div itemprop="location" itemscope itemtype="https://schema.org/Place">
    <span itemprop="name">Bangalore Convention Center</span>
    <div itemprop="address" itemscope itemtype="https://schema.org/PostalAddress">
      <span itemprop="streetAddress">123 MG Road</span>,
      <span itemprop="addressLocality">Bengaluru</span>
    </div>
  </div>
</div>

12.2.4 Article & BlogPosting schemas

Articles in research & blogs Concept

Article and BlogPosting capture news, blog entries, and research notes. They are central to academic and technical publishing online.

Example: BlogPosting in JSON-LD (recommended by Google) JSON-LD

Even though this module focuses on microdata, in practice JSON-LD is often preferred. You should understand both. 


<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "BlogPosting",
  "headline": "Understanding Microdata and JSON-LD",

    "@type": "Person",
    "name": "Irfan"
  },
  "datePublished": "2025-11-24",
  "mainEntityOfPage": {
    "@type": "WebPage",
    "@id": "https://example.com/microdata-vs-jsonld"
  }
}
</script>
12.3

Testing & Validation Quality Assurance

Structured data only helps if it is valid and interpretable by consumers. Validation tools ensure that your microdata/JSON-LD matches Schema.org rules and search engine requirements.

12.3.1 Google’s Rich Results Test

Purpose of Rich Results Test Tool
  • Validates whether your structured data qualifies for enhanced results (stars, carousels, event cards, etc.).
  • Can test via URL or code snippet.
  • Shows which rich result types are detected, warnings, and errors.
Conceptual “algorithm” of validation (pseudocode) Pseudocode 

Given: HTML page P

1. Extract all structured data blocks:
   - Microdata
   - RDFa
   - JSON-LD

2. For each block B:
   a. Parse B into an internal graph representation.
   b. Identify the main entity type T (e.g., Product, Event).
   c. Look up schema specification for T:
      - Required properties R
      - Recommended properties S

3. Check constraints:
   a. Are all required properties in R present?
   b. Are their value types consistent? (date, URL, text, number)
   c. Are there any unknown or deprecated properties?

4. Aggregate results:
   - If any required property missing ⇒ Error
   - If recommended properties missing ⇒ Warning
   - Otherwise ⇒ Eligible for rich results (if type supported)

12.3.2 Schema markup validation

Multiple validators Tools
  • Google Rich Results Test: Focuses on rich result eligibility.
  • Schema.org validator / structured data testing tools: Focus on syntax and Schema.org vocabulary correctness.
  • Search Console: Gives ongoing reports for your entire site, surfacing aggregate issues over time.

In a production pipeline, you would integrate at least one validator into CI, so that invalid structured data fails the build or raises alerts.

12.3.3 SEO benefits of structured data

How structured data affects SEO Concept
  • Rich snippets: Ratings, price, availability, event dates, FAQ accordions, how-to steps, breadcrumbs, etc., directly in SERPs.
  • Improved CTR: Richer visual appearance leads to higher click-through rates, even if ranking position is unchanged.
  • Better disambiguation: Helps search engines distinguish between entities with similar names (e.g., the researcher “Irfan” vs a brand or product with the same name).
  • Knowledge graphs: Clean structured data contributes to entity understanding, which influences knowledge panels, “People also ask”, and assistant answers.
Wrap-up

Checklist: Microdata & structured data Self-Audit

Quick self-check Checklist
  • Have you identified the main entities on the page (Person, Organization, Product, Event, Article, etc.)?
  • Does each entity have itemscope and a correct itemtype (Schema.org URL)?
  • Are required properties for each type present and correctly typed?
  • Are you using itemprop only with valid property names for the given type?
  • Have you validated the markup using at least one structured data testing tool?
  • For SEO-critical pages, have you checked which rich result types your markup enables?
  • Do your code examples render cleanly, with tags like <ul> and <li> visible and no stray class="hl-tag" artifacts?

If you can answer “yes” to these, your structured data is not just syntactically valid but semantically meaningful, ready for integration into modern search and knowledge systems.
