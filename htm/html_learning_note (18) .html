13.1

Accessibility Fundamentals Foundations

Accessibility (A11y) is the practice of designing digital experiences that are usable by everyone, including people with disabilities. WCAG (Web Content Accessibility Guidelines) provide the global standard for inclusive, perceivable, operable, understandable, and robust interfaces.

Understanding disabilities and assistive technologies Concept

Disabilities can be permanent (blindness), temporary (arm injury), or situational (holding a baby, driving). Designing inclusively benefits all users.

  • Visual: blindness, low vision, color blindness
  • Auditory: deafness, hearing loss
  • Motor: tremors, paralysis, limited fine motor control
  • Cognitive: dyslexia, ADHD, memory impairments

Users interact via assistive technologies:

  • Screen readers: NVDA, JAWS, VoiceOver
  • Screen magnifiers
  • Speech recognition: Dragon NaturallySpeaking
  • Switch access / alternate input devices

Your HTML must expose proper semantics so assistive tech can interpret structure, landmarks, roles, labels, and states.

13.1.2 WCAG 2.1 Guidelines Overview

The four WCAG principles (POUR) Concept

WCAG 2.1 organizes accessibility requirements under four principles:

  • Perceivable — information must be presented in ways users can perceive
  • Operable — interface elements must be usable via different inputs
  • Understandable — content must be readable, predictable, consistent
  • Robust — compatible with assistive technologies

WCAG is divided into:

  • A (minimum compliance)
  • AA (industry standard — most required)
  • AAA (highest, not always practical)

WCAG is intentionally *technology-agnostic* — it tells “what must be true” rather than “how to code it”. 


Pseudocode summary of WCAG checks:

For each interface component C:
    Check if C is perceivable:
        - Has text alternative?
        - Has sufficient color contrast?

    Check if C is operable:
        - Keyboard-accessible?
        - Focus visible?
        - Timing adjustable?

    Check if C is understandable:
        - Consistent navigation?
        - Predictable behavior?

    Check if C is robust:
        - Correct HTML semantics?
        - ARIA states conveyed?

13.1.3 Accessibility Testing Tools

Manual + automated testing tools Tools
  • Axe DevTools (browser extension)
  • Lighthouse A11y audits
  • NVDA (free Windows screen reader)
  • VoiceOver (Mac/iOS)
  • WAVE Tool
  • Keyboard-only testing (Tab, Shift+Tab, Enter, Space)

Accessibility testing workflow:

1. Automated scan (Lighthouse, Axe)
2. Keyboard navigation test
3. Screen reader test
4. Color contrast check
5. ARIA validation
6. Manual review of headings, landmarks, labels
13.2

Semantic HTML for Accessibility Structure & Behavior

Semantic HTML is the foundation of accessibility. If the underlying DOM correctly expresses document structure, landmarks, relationships, and control types, assistive technologies can interpret your UI without hacks.

13.2.1 Proper heading structure

Why heading hierarchy matters Concept

Screen readers allow users to navigate by headings, like a “table of contents overlay”. If your headings are logical, users can skim efficiently.

  • Exactly one logical top-level heading (often <h1>)
  • Subtopics use <h2>, sub-subtopics use <h3>, etc.
  • Do not skip levels purely for visual styling.
Example: good vs bad heading hierarchy Example 

<!-- ✅ Good: logical outline -->
<main>
  <h1>Accessibility Documentation</h1>

  <section>
    <h2>Getting Started</h2>
    <h3>Keyboard Navigation</h3>
    <h3>Screen Reader Basics</h3>
  </section>

  <section>
    <h2>Advanced Topics</h2>
    <h3>ARIA Roles</h3>
    <h3>Live Regions</h3>
  </section>
</main>

<!-- ❌ Bad: skipping levels and visual-only headings -->
<main>
  <h1>Accessibility Documentation</h1>
  <h4>Getting Started</h4> <!-- visually smaller but structurally wrong -->
  <h6>Keyboard Navigation</h6>
</main>

Visual size should be controlled via CSS, not by abusing heading levels.

13.2.2 Landmark roles and navigation

Native landmarks & ARIA roles Concept

Landmarks let screen reader users jump to high-level regions: navigation, main content, search, etc.

  • Native landmarks: <header>, <nav>, <main>, <aside>, <footer>
  • Complementary roles: role="main", role="navigation", role="banner", role="contentinfo"
  • Use ARIA landmarks when you cannot use native elements (e.g. legacy markup, div-heavy layouts).
Example: landmarks with skip link Example 

<body>
  <a href="#main-content" class="skip-link">
    Skip to main content
  </a>

  <header>
    <h1>Accessible Application</h1>
  </header>

  <nav aria-label="Primary navigation">
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/docs">Docs</a></li>
    </ul>
  </nav>

  <main id="main-content">
    <h2>Dashboard</h2>
    <!-- Page-specific content -->
  </main>

  <footer>
    &copy; 2025 Irfan Corp
  </footer>
</body>

Many screen readers provide a “Landmarks” or “Regions” rotor allowing users to jump between these sections.

13.2.3 Focus management

Keyboard focus rules Concept

Many users rely on keyboard-only navigation. Focus must move in a predictable, logical order, and never get “trapped”.

  • All interactive elements must be focusable (links, buttons, inputs).
  • Avoid using tabindex > 0 (it breaks natural order).
  • Ensure visible focus styling (WCAG 2.4.7).
  • When opening dialogs, move focus inside the dialog and restore it on close.
Example: managing focus in a dialog Example + JS 

<button id="openDialog">Open Settings</button>

<div id="dialog" role="dialog" aria-modal="true"
     aria-labelledby="dialog-title" hidden>
  <h2 id="dialog-title">Settings</h2>
  <button id="closeDialog">Close</button>
</div>

<script>
  const openBtn = document.getElementById('openDialog');
  const closeBtn = document.getElementById('closeDialog');
  const dialog = document.getElementById('dialog');
  let lastFocusedElement = null;

  openBtn.addEventListener('click', () => {
    lastFocusedElement = document.activeElement;
    dialog.hidden = false;
    closeBtn.focus(); // move focus into dialog
  });

  closeBtn.addEventListener('click', () => {
    dialog.hidden = true;
    if (lastFocusedElement) lastFocusedElement.focus(); // restore focus
  });
</script>

This pattern prevents keyboard users from “losing themselves” when dialogs open and close.

13.2.4 Screen reader considerations

How screen readers interpret your HTML Concept

Screen readers create a virtual accessibility tree from the DOM, influenced by:

  • Element names (role)
  • Relationships (labels, descriptions, ownership)
  • States and properties (expanded, checked, disabled)

Avoid hiding important content visually-only or screen-reader-only without clear reason.

Example: visually hidden text for SR users Example 

<style>
  .sr-only {
    position: absolute;
    width: 1px;
    height: 1px;
    padding: 0;
    margin: -1px;
    overflow: hidden;
    clip: rect(0, 0, 0, 0);
    white-space: nowrap;
    border: 0;
  }
</style>

<button>
  <span class="sr-only">Close notification</span>
  <span aria-hidden="true">×</span>
</button>

Screen reader users hear “Close notification”, while sighted users see only the × icon.

13.3

ARIA Attributes Enhancing Semantics

Accessible Rich Internet Applications (ARIA) is a specification that lets you describe roles, states, and properties for complex UI components. ARIA must be used carefully: “No ARIA is better than bad ARIA.”

13.3.1 When to use ARIA

Use native first, ARIA second Guideline
  • Prefer native HTML elements (<button>, <a>, <input>) because they come with built-in semantics and keyboard behavior.
  • Use ARIA when:
    • You build custom components (tabs, accordions, menus).
    • You need to expose dynamic state (expanded, selected, busy).
    • You want to label or describe controls beyond what visible text allows.
  • Never use ARIA to “fake” a button on a <div> without also implementing keyboard support. 

ARIA decision rule:

If native element exists that expresses the same semantics:
    Use native element.
Else:
    Use ARIA role + keyboard handling + state management.

13.3.2 aria-label and aria-labelledby

Naming controls for assistive tech Concept
  • aria-label: string label applied directly on the element.
  • aria-labelledby: references one or more elements whose text forms the label.
  • Prefer aria-labelledby when there is visible text on the screen.
Example: icon-only button with aria-label Example 

<button type="button" aria-label="Open navigation menu">
  <span aria-hidden="true">☰</span>
</button>

Screen reader users hear “Open navigation menu”, not “☰”.

Example: using aria-labelledby with visible text Example 

<h2 id="profile-heading">User Profile</h2>

<section aria-labelledby="profile-heading">
  <p>Profile details and preferences.</p>
</section>

The section is labeled by the heading’s text. This pattern is also common with dialogs, groups, and complex widgets.

13.3.3 aria-describedby

Providing extra descriptive context Concept

aria-describedby points to elements that offer additional description, not the main label. Common for form hints, error messages, and detailed instructions.

Example: input with description and error Example 

<label for="email">Email address</label>
<input id="email" name="email" type="email"
       aria-describedby="email-hint email-error" />

<p id="email-hint">
  We will never share your email with anyone else.
</p>

<p id="email-error" role="alert">
  Please enter a valid email address.
</p>

A screen reader will announce: “Email address, edit, We will never share your email…, Please enter a valid email address”.

13.3.4 Live regions and dynamic content

Announcing dynamic changes Concept

Live regions let you notify screen reader users when content updates without moving focus.

  • aria-live="polite" — announce when convenient
  • aria-live="assertive" — interrupt immediately (use sparingly)
  • role="status" and role="alert" imply live behavior
Example: async status messages Example + JS 

<button id="saveBtn">Save changes</button>

<div id="status" role="status" aria-live="polite">
  <!-- Status messages will appear here -->
</div>

<script>
  const saveBtn = document.getElementById('saveBtn');
  const status = document.getElementById('status');

  saveBtn.addEventListener('click', () => {
    status.textContent = 'Saving...';

    setTimeout(() => {
      status.textContent = 'Changes saved successfully.';
    }, 1500);
  });
</script>

Sighted users see the text change; screen reader users hear “Saving…” then “Changes saved successfully.”

13.3.5 ARIA states and properties

Common ARIA states Reference
  • aria-expanded — expanded/collapsed state of a control
  • aria-pressed — pressed state of toggle buttons
  • aria-selected — selected state in listbox, tabs, etc.
  • aria-hidden — whether the element is exposed to the accessibility tree
  • aria-disabled — unavailable but visible state
Example: custom accordion with aria-expanded Example + JS 

<button class="accordion-trigger"
        aria-expanded="false"
        aria-controls="panel-1"
        id="accordion-1">
  Accessibility FAQ
</button>

<div id="panel-1" role="region"
     aria-labelledby="accordion-1" hidden>
  <p>Here goes the answer about accessibility.</p>
</div>

<script>
  const trigger = document.querySelector('.accordion-trigger');
  const panel = document.getElementById('panel-1');

  trigger.addEventListener('click', () => {
    const isExpanded = trigger.getAttribute('aria-expanded') === 'true';
    trigger.setAttribute('aria-expanded', String(!isExpanded));
    panel.hidden = isExpanded; // hide if it was expanded, show otherwise
  });
</script>

This pattern makes the state of the accordion accessible both visually and via assistive technologies.

Wrap-up

Checklist: Accessibility (WCAG + ARIA) Self-Audit

Quick self-check Checklist
  • Do all images have appropriate alt text?
  • Is your heading structure logical and nested correctly?
  • Are landmarks (<header>, <nav>, <main>, <footer>, etc.) used properly?
  • Can every interactive component be reached and used via keyboard only?
  • Is focus visible and managed when dialogs/overlays appear?
  • Do custom components use appropriate ARIA roles, states, and properties?
  • Have you tested with at least one screen reader and one automated tool?

If you can answer “yes” to these, you are aligning with WCAG principles and building truly inclusive experiences.
