Module 8 · Forms – Foundations

HTML Forms: Capturing User Input Correctly and Safely

Forms are the bridge between your users and your application logic. Every login screen, signup page, search bar, feedback form, and payment checkout is built on HTML forms. In this module, you will learn how to structure forms properly, use basic input types, and keep them accessible and user-friendly.

Focus: Semantic, accessible form foundations <form> and attributes action & method built-in validation text & password inputs email, URL, number, range checkboxes & radios labels & placeholders error feedback
Executive Overview of Module 8
High-level map before details
  • 8.1 Form Structure: Learn what the <form> element does, how action and method control submissions, and how HTML5 provides built-in validation hooks.
  • 8.2 Basic Input Types: Understand common input types like text, password, email, URL, number, range, checkboxes, and radios, and when to use each one.
  • 8.3 Labels & Accessibility: Discover how to connect labels and inputs, when placeholders are helpful (and when they are harmful), how to mark required fields, and how to provide clear error messages.
Big idea: A good form is not just HTML tags. It is a combination of structure, semantics, usability, accessibility, and validation. You should always design forms with both humans and machines in mind.

8.1 Form Structure <form> as the container

At the core of any HTML form is the <form> element. It defines a region of the page where user input is collected and specifies what happens when the user submits that data.

8.1.1 <form> element and attributes
Form container and basic attributes

The <form> element wraps all controls that belong to one submission unit (e.g., login, signup, feedback). Key attributes:

  • action — where the form data should be sent.
  • method — how the data is sent (HTTP method, usually GET or POST).
  • novalidate — disable built-in browser validation (optional).
  • autocomplete — allow or disable browser auto-fill (“on” / “off”).

Basic example: simple contact form (structure only).

<form action="/contact" method="post">
    <div>
        <label for="name">Name</label>
        <input type="text" id="name" name="name">
    </div>

    <div>
        <label for="email">Email</label>
        <input type="email" id="email" name="email">
    </div>

    <button type="submit">Send</button>
</form>
In real applications, the action often points to a server route or is left empty when JavaScript handles the submission (using event.preventDefault()).
8.1.2 Form action and method
GET vs POST and URLs

The action attribute defines the target URL; the method attribute defines the HTTP method.

  • method="get" — sends data in the URL query string (visible in the address bar).
  • method="post" — sends data in the request body (not in the URL).

Example: search form (GET is appropriate for non-sensitive queries).

<form action="/search" method="get">
    <label for="q">Search</label>
    <input type="text" id="q" name="q">
    <button type="submit">Search</button>
</form>

If the user enters “html forms” and submits, the resulting URL might be: 

/search?q=html+forms

Example: login form (POST is preferred for credentials).

<form action="/login" method="post">
    <label for="email">Email</label>
    <input type="email" id="email" name="email">

    <label for="password">Password</label>
    <input type="password" id="password" name="password">

    <button type="submit">Sign in</button>
</form>
Rule of thumb Security-relevant
Use GET for safe, idempotent operations (search, filter, view). Use POST for actions that create or update data, or involve sensitive information like passwords.
8.1.3 Form validation overview
HTML5 constraints and JavaScript

Validation ensures that the data submitted by the user is complete and correctly formatted. There are three main layers:

  • HTML5 constraint validation: attributes like required, min, max, pattern, type="email", etc.
  • JavaScript client-side validation: custom checks before sending data to the server (e.g., password strength, matching fields).
  • Server-side validation: final and authoritative validation on the server (must never be skipped).

Simple HTML5 validation example:

<form action="/subscribe" method="post">
    <label for="email">Email</label>
    <input
        type="email"
        id="email"
        name="email"
        required>

    <button type="submit">Subscribe</button>
</form>
If the user submits this form with an empty email or an invalid address, the browser will show a default validation message, even without any JavaScript. However, you should still validate on the server.

8.2 Basic Input Types Collecting different kinds of data

HTML provides many specialized input types. Choosing the right type improves usability, mobile keyboard behavior, and validation quality, often with almost no extra effort.

8.2.1 Text inputs (<input type="text">)
Free-form single-line text

A text input is the most generic single-line input. Use it for names, titles, short labels, or any content that does not have a more specific type. 

<form>
    <label for="full-name">Full name</label>
    <input
        type="text"
        id="full-name"
        name="full_name"
        autocomplete="name"
        required>
</form>
  • autocomplete="name" helps browsers and password managers auto-fill the value.
  • Always connect text inputs to labels (see section 8.3.1).
8.2.2 Password fields
Masking sensitive input

Password fields behave like text inputs, but characters are visually masked. They do not provide intrinsic security (data still travels in plain text unless you use HTTPS), but they protect against shoulder-surfing. 

<form action="/login" method="post">
    <label for="email">Email</label>
    <input
        type="email"
        id="email"
        name="email"
        autocomplete="email"
        required>

    <label for="password">Password</label>
    <input
        type="password"
        id="password"
        name="password"
        autocomplete="current-password"
        required>

    <button type="submit">Sign in</button>
</form>
For signup forms, use autocomplete="new-password" so browsers and password managers can suggest strong unique passwords.
8.2.3 Email and URL inputs
Semantic formats with built-in checks

Using dedicated types like type="email" and type="url" has several benefits:

  • Browsers can show specialized keyboards on mobile (e.g., “@” button for email).
  • Built-in validation checks basic format before submission.
  • Assistive technologies can interpret the field more intelligently.

Email input:

<label for="email">Email address</label>
<input
    type="email"
    id="email"
    name="email"
    required
    placeholder="you@example.com">

URL input:

<label for="website">Website</label>
<input
    type="url"
    id="website"
    name="website"
    placeholder="https://example.com">
Browser validation checks that an email at least contains something like user@example.com, and that a URL looks like a valid address. You still must validate more thoroughly on the server.
8.2.4 Number and range inputs
Numeric values and sliders

When you expect numeric values, use type="number". For approximate values or “scale” style input, use type="range".

Number input example:

<label for="age">Age</label>
<input
    type="number"
    id="age"
    name="age"
    min="0"
    max="120"
    step="1"
    required>
  • min / max — numeric constraints.
  • step — increment/decrement step (e.g., 0.5 or 10).

Range input example:

<label for="satisfaction">Satisfaction (1–10)</label>
<input
    type="range"
    id="satisfaction"
    name="satisfaction"
    min="1"
    max="10"
    step="1"
    value="5">
Range inputs are nice for sliders, but the actual numeric value should usually be shown alongside (e.g., in a <span> updated via JavaScript) so users know the exact selection.
8.2.5 Checkboxes and radio buttons
Multiple vs single selection

Checkboxes and radios represent choices, but they behave differently:

  • Checkboxes: allow multiple selections; each is independent.
  • Radio buttons: allow exactly one selection within a named group.

Checkbox example (multiple topics):

<fieldset>
    <legend>Topics you are interested in</legend>

    <div>
        <input
            type="checkbox"
            id="topic-html"
            name="topics"
            value="html">
        <label for="topic-html">HTML</label>
    </div>

    <div>
        <input
            type="checkbox"
            id="topic-css"
            name="topics"
            value="css">
        <label for="topic-css">CSS</label>
    </div>

    <div>
        <input
            type="checkbox"
            id="topic-js"
            name="topics"
            value="javascript">
        <label for="topic-js">JavaScript</label>
    </div>
</fieldset>

Radio example (one plan only):

<fieldset>
    <legend>Choose a plan</legend>

    <div>
        <input
            type="radio"
            id="plan-basic"
            name="plan"
            value="basic"
            checked>
        <label for="plan-basic">Basic</label>
    </div>

    <div>
        <input
            type="radio"
            id="plan-pro"
            name="plan"
            value="pro">
        <label for="plan-pro">Pro</label>
    </div>

    <div>
        <input
            type="radio"
            id="plan-enterprise"
            name="plan"
            value="enterprise">
        <label for="plan-enterprise">Enterprise</label>
    </div>
</fieldset>
Using <fieldset> and <legend> groups related options semantically, which is very helpful for screen readers.

8.3 Form Labels and Accessibility Helping users understand and succeed

Forms are one of the most common sources of frustration on the web. Clear labels, usable placeholders, required indicators, and good error messages can transform a form from painful to pleasant — and are critical for accessibility.

8.3.1 <label> element association
for / id pattern

Every interactive control should have a label. The most common pattern is:

  • Give the input an id.
  • Give the label a matching for attribute.
<label for="username">Username</label>
<input type="text" id="username" name="username" required>

Clicking the label focuses the input, which helps all users (especially those with low motor control) and provides an accessible name for screen readers.

Another pattern wraps the input inside the label (often used for checkboxes and radios): 

<label>
    <input type="checkbox" name="subscribe" value="yes">
    Subscribe to newsletter
</label>
Both patterns are valid; just ensure that every input has exactly one clear accessible label.
8.3.2 Placeholder vs label
Do not use placeholder as label

A placeholder is a hint displayed inside the input when it is empty. It is not a replacement for a real label.

Correct usage (label + placeholder):

<label for="email">Email</label>
<input
    type="email"
    id="email"
    name="email"
    placeholder="you@example.com"
    required>
  • Label: permanent, always visible, describes what the field represents.
  • Placeholder: optional, gives a hint about the expected format or example.

Bad pattern — using only placeholder, no label:

<!-- Bad: no label, only placeholder -->
<input
    type="email"
    name="email"
    placeholder="Email">
Placeholders can disappear as soon as the user starts typing, and their text often has low contrast. This makes forms hard to use for many people. Always keep a proper label.
8.3.3 Required fields and validation
Marking and enforcing constraints

Required fields should be:

  • Marked visually (e.g., with an asterisk * and a note).
  • Marked semantically using the required attribute.
  • Enforced by server-side validation as the final authority.

Example: basic signup form with required fields.

<form action="/signup" method="post">
    <div>
        <label for="signup-name">Full name <span aria-hidden="true">*</span></label>
        <input
            type="text"
            id="signup-name"
            name="name"
            required
            aria-required="true">
    </div>

    <div>
        <label for="signup-email">Email <span aria-hidden="true">*</span></label>
        <input
            type="email"
            id="signup-email"
            name="email"
            required
            aria-required="true">
    </div>

    <button type="submit">Create account</button>
</form>
aria-required="true" is not strictly necessary if you use required, but it can improve compatibility with some assistive technologies.
8.3.4 Error handling and feedback
Helping users fix problems

Good error handling answers three questions clearly:

  1. Where is the error?
  2. What is wrong?
  3. How can the user fix it?

Pattern for accessible error messages:

  • Place an error message near the field.
  • Use aria-describedby to link the field to the message.
  • Optionally use role="alert" for live announcements.
<form action="/login" method="post">
    <div>
        <label for="login-email">Email</label>
        <input
            type="email"
            id="login-email"
            name="email"
            required
            aria-describedby="login-email-error">

        <p id="login-email-error" class="error-message">
            Please enter a valid email address.
        </p>
    </div>

    <div>
        <label for="login-password">Password</label>
        <input
            type="password"
            id="login-password"
            name="password"
            required>
    </div>

    <button type="submit">Sign in</button>
</form>

Conceptual CSS example for error state:

.error-message {
    color: #f87171; /* soft red */
    font-size: 0.8rem;
    margin-top: 0.15rem;
}

input[aria-describedby="login-email-error"] {
    border-color: #f97316; /* orange-ish border hint */
}
Instead of generic messages like “Invalid input”, always tell users exactly what went wrong: “Password must be at least 8 characters and include a number.” This is good UX and reduces frustration.
Practice ideas:
  • Build a small login form and a separate signup form using proper labels and input types.
  • Create a survey form using text, email, number, range, checkboxes, and radios together.
  • Add HTML5 validation (required, type, min/max) and observe how browsers respond.
  • Design at least one error state and wire it with aria-describedby.
