Media Queries

Create responsive experiences that adapt to any device, screen size, or user preference

11.1 Media Query Fundamentals

The foundation of responsive design — understanding how to target different devices and conditions.

Media types and features

Media queries allow you to apply styles based on device characteristics.


/* Basic syntax */
@media media-type and (media-feature) {
    /* CSS rules */
}

/* Common media types */
@media screen {
    /* Styles for screens */
}

@media print {
    /* Styles for printing */
}

@media handheld {
    /* Legacy mobile devices */
}

/* Common media features */
@media (min-width: 768px) {
    /* Styles for screens at least 768px wide */
}

@media (max-width: 1024px) {
    /* Styles for screens up to 1024px wide */
}

@media (orientation: portrait) {
    /* Styles when device is in portrait mode */
}

@media (prefers-color-scheme: dark) {
    /* Styles when user prefers dark mode */
}

This is a basic media query demo. Resize your browser window to see how styles change!

Media Types:

Media Type Description Use Case
all Applies to all devices Default if no type specified
screen Computer screens, tablets, smartphones ✅ Most common for web applications
print Printers and print preview Optimize printed documents
speech Screen readers and speech synthesizers Accessibility enhancements
handheld Legacy mobile devices Deprecated — use screen instead

Media Features:

  • width, height: Viewport dimensions
  • min-width, max-width: Minimum/maximum viewport width
  • orientation: Portrait or landscape
  • aspect-ratio: Width-to-height ratio
  • resolution: Pixel density
  • prefers-color-scheme: User's preferred color scheme
  • prefers-reduced-motion: User prefers less motion
  • hover: Device supports hover
  • pointer: Device has precise pointing mechanism

Pro Tip: Always use screen for web applications — it's the most reliable and widely supported.

Viewport meta tag importance

The essential HTML tag that controls how your page is displayed on mobile devices.


<meta name="viewport" content="width=device-width, initial-scale=1.0">

Without viewport meta tag:

On mobile devices, browsers render pages at a typical desktop screen width (usually 980px), then shrink them to fit. This makes text too small to read without zooming.

With viewport meta tag:

Browser sets the viewport width to match the device width, so content renders properly at native scale. Text is readable without zooming.

What the viewport tag does:

  • width=device-width: Sets the viewport width to match the device's screen width
  • initial-scale=1.0: Sets the initial zoom level to 1:1
  • maximum-scale=1.0: Prevents users from zooming (use cautiously)
  • user-scalable=no: Disables zooming entirely (accessibility violation)

Why it matters:

  • Without it, media queries won't work as expected on mobile
  • Content appears tiny and unreadable on phones
  • Breakpoints trigger at wrong widths
  • Bad user experience across mobile devices

Best Practice: Always include this tag in the <head> of every webpage:


<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <!-- Your other head elements -->
</head>

Accessibility Warning: Never use user-scalable=no or maximum-scale=1.0 — it violates accessibility guidelines and prevents users from zooming when needed.

Mobile-first vs desktop-first approaches

Two philosophies for structuring your responsive CSS.


/* Mobile-first approach (recommended) */
/* Base styles for mobile */
.container {
    width: 100%;
    padding: 10px;
}

/* Desktop enhancements */
@media (min-width: 768px) {
    .container {
        width: 750px;
        margin: 0 auto;
        padding: 20px;
    }
}

/* Desktop-first approach (legacy) */
/* Base styles for desktop */
.container {
    width: 1200px;
    margin: 0 auto;
    padding: 30px;
}

/* Mobile adaptations */
@media (max-width: 767px) {
    .container {
        width: 100%;
        padding: 10px;
    }
}

Mobile-First

Start with simple mobile styles → Add complexity for larger screens

  • Smaller CSS file size
  • Faster loading on mobile
  • Progressive enhancement
  • Easier maintenance

Desktop-First

Start with complex desktop styles → Remove complexity for smaller screens

  • Larger CSS file size
  • Slower loading on mobile
  • Overridden styles
  • Harder to maintain

Why mobile-first wins:

  1. Performance: Mobile users have slower connections — send less CSS initially
  2. Simplicity: Start with the simplest layout, then enhance
  3. Progressive Enhancement: Build upon a solid foundation
  4. Maintainability: Easier to understand and modify
  5. Modern Standard: Industry best practice since 2015

Real-World Example:


/* Mobile-first: Simple base */
.button {
    display: block;
    width: 100%;
    padding: 15px;
    font-size: 1rem;
    border-radius: 5px;
}

/* Tablet enhancement */
@media (min-width: 768px) {
    .button {
        display: inline-block;
        width: auto;
        padding: 12px 24px;
        font-size: 1.1rem;
        border-radius: 8px;
    }
}

/* Desktop enhancement */
@media (min-width: 1024px) {
    .button {
        padding: 15px 30px;
        font-size: 1.2rem;
        border-radius: 10px;
    }
}

Pro Tip: Use min-width queries for mobile-first — they're more intuitive and scalable.

Mobile-First Workflow:

  1. Design for smallest screen first
  2. Write base CSS for mobile
  3. Add breakpoints only when necessary
  4. Test on real devices
  5. Refine for larger screens

11.2 Advanced Media Query Features

Target specific device capabilities and user preferences beyond just screen size.

Resolution and pixel density queries

Target high-resolution displays like Retina screens and modern 4K monitors.


/* Target high-DPI displays */
@media (-webkit-min-device-pixel-ratio: 2),
       (min-resolution: 192dpi) {
    .logo {
        background-image: url('logo@2x.png');
    }
    
    .icon {
        background-image: url('icon@2x.png');
    }
}

/* For 3x resolution (modern phones) */
@media (min-resolution: 288dpi) {
    .hero-image {
        background-image: url('hero@3x.jpg');
    }
}

/* Using dppx unit (device pixels per px) */
@media (min-resolution: 2dppx) {
    .retina-image {
        background-image: url('image@2x.webp');
    }
}

/* Print-specific high resolution */
@media print and (min-resolution: 300dpi) {
    .print-image {
        image-rendering: crisp-edges;
    }
}

High-resolution display detection

This content appears differently on retina displays

Pixel Density Units:

Unit Description Example
dpi Dots per inch 192dpi = 2x pixel density
dpcm Dots per centimeter 75dpcm = 2x pixel density
dppx Device pixels per CSS pixel 2dppx = 2x pixel density

Conversion Guide:

  • 1dppx = 96dpi
  • 2dppx = 192dpi
  • 3dppx = 288dpi

Best Practices:

  1. Use dppx instead of dpi — it's more accurate
  2. Always provide fallback images for lower-resolution displays
  3. Use modern formats like WebP for better compression
  4. Consider using <picture> element for art direction

Performance Tip: Don't serve massive images to low-end devices — use media queries to deliver appropriately sized assets.

Orientation and aspect ratio

Adapt layouts based on device orientation and screen proportions.


/* Orientation queries */
@media (orientation: portrait) {
    /* Styles for portrait mode */
    .sidebar {
        display: none;
    }
    
    .main-content {
        width: 100%;
    }
}

@media (orientation: landscape) {
    /* Styles for landscape mode */
    .sidebar {
        display: block;
        width: 250px;
    }
    
    .main-content {
        width: calc(100% - 250px);
    }
}

/* Aspect ratio queries */
@media (aspect-ratio: 16/9) {
    /* Styles for 16:9 screens */
    .video-player {
        width: 100%;
        height: 56.25%; /* 9/16 = 0.5625 */
    }
}

@media (min-aspect-ratio: 3/2) {
    /* Styles for wider screens */
    .grid {
        grid-template-columns: repeat(4, 1fr);
    }
}

@media (max-aspect-ratio: 2/1) {
    /* Styles for narrower screens */
    .grid {
        grid-template-columns: repeat(2, 1fr);
    }
}

Aspect Ratio Detection

Rotate your device to see different layouts!

Common Aspect Ratios:

  • 4/3 — Traditional TV (1.33)
  • 16/9 — Standard widescreen (1.78)
  • 21/9 — Ultra-wide monitor (2.33)
  • 1/1 — Square (1.0)
  • 3/4 — Portrait smartphone (0.75)

Practical Use Cases:

  • Portrait: Hide sidebars, stack content vertically
  • Landscape: Show sidebars, use multi-column layouts
  • Wide screens: Increase number of columns
  • Narrow screens: Reduce number of columns
  • Video players: Maintain correct aspect ratio

Pro Tip: Combine orientation with width queries for precise control:


/* Phone in portrait */
@media (max-width: 767px) and (orientation: portrait) {
    .content {
        padding: 15px;
    }
}

/* Phone in landscape */
@media (max-width: 767px) and (orientation: landscape) {
    .content {
        padding: 10px;
    }
}

/* Tablet in portrait */
@media (min-width: 768px) and (max-width: 1023px) and (orientation: portrait) {
    .sidebar {
        display: none;
    }
}
Color gamut and color index

Target devices with advanced color capabilities for richer visual experiences.


/* Color gamut queries */
@media (color-gamut: srgb) {
    /* Standard colors — most devices */
    .gradient {
        background: linear-gradient(to right, #00d1ff, #00a8ff);
    }
}

@media (color-gamut: p3) {
    /* Wide color gamut — modern displays */
    .gradient {
        background: linear-gradient(to right, rgb(0, 209, 255), rgb(0, 168, 255));
    }
}

@media (color-gamut: rec2020) {
    /* Ultra-wide color gamut — professional displays */
    .gradient {
        background: linear-gradient(to right, rgb(0, 209, 255), rgb(0, 168, 255));
        filter: brightness(1.1);
    }
}

/* Color index queries */
@media (monochrome: 0) {
    /* Color displays */
    .logo {
        filter: hue-rotate(15deg);
    }
}

@media (monochrome: 1) {
    /* Grayscale displays */
    .logo {
        filter: grayscale(100%);
    }
}

Color Gamut Detection

Colors appear differently on high-end displays

Color Gamuts Explained:

Color Gamut Range Devices
srgb Standard RGB Most computers, phones, TVs
p3 Apple Display P3 iPhones, iPads, MacBooks, modern Android
rec2020 Ultra HD Professional monitors, high-end TVs

Color Index:

  • monochrome: 0 — Color display
  • monochrome: 1 — Grayscale display
  • monochrome: 2 — Monochrome with 2 levels

When to use:

  • Use p3 for premium experiences on Apple and modern Android devices
  • Use monochrome for accessibility on grayscale displays
  • Use rec2020 for professional video editing interfaces

Pro Tip: Always provide a fallback for standard color gamut — not all devices support wide gamut!


/* Progressive enhancement for color */
.gradient {
    /* Fallback for sRGB */
    background: linear-gradient(to right, #00d1ff, #00a8ff);
}

/* Enhanced for P3 */
@media (color-gamut: p3) {
    .gradient {
        background: linear-gradient(to right, rgb(0, 209, 255), rgb(0, 168, 255));
    }

Prefers-color-scheme and user preferences

Respect user preferences for accessibility and comfort.


/* Dark mode */
@media (prefers-color-scheme: dark) {
    :root {
        --bg-color: #0a192f;
        --text-color: #e6f1ff;
        --accent-color: #ffb74d;
    }
    
    body {
        background-color: var(--bg-color);
        color: var(--text-color);
    }
    
    .card {
        background-color: rgba(13, 27, 42, 0.7);
        border: 1px solid rgba(0, 209, 255, 0.2);
    }
}

/* Light mode */
@media (prefers-color-scheme: light) {
    :root {
        --bg-color: #ffffff;
        --text-color: #333333;
        --accent-color: #ffa726;
    }
    
    body {
        background-color: var(--bg-color);
        color: var(--text-color);
    }
    
    .card {
        background-color: #f8f9fa;
        border: 1px solid #dee2e6;
    }
}

/* No preference */
@media (prefers-color-scheme: no-preference) {
    /* Default theme */
    :root {
        --bg-color: #ffffff;
        --text-color: #333333;
        --accent-color: #ffa726;
    }
}

/* Reduced motion */
@media (prefers-reduced-motion: reduce) {
    * {
        animation-duration: 0.01ms !important;
        animation-iteration-count: 1 !important;
        transition-duration: 0.01ms !important;
    }
}

/* High contrast */
@media (prefers-contrast: high) {
    body {
        background-color: #000;
        color: #fff;
    }
    
    .card {
        border: 3px solid #fff;
    }
}

/* Forced colors */
@media (forced-colors: active) {
    body {
        background-color: Canvas;
        color: CanvasText;
    }
    
    .card {
        border: 1px solid ButtonText;
    }
}

Your system settings determine these styles!

Key Preference Queries:

Query Description Importance
prefers-color-scheme User's preferred light/dark mode ✅ Essential for accessibility
prefers-reduced-motion User prefers less animation ✅ Critical for vestibular disorders
prefers-contrast User prefers high contrast ✅ Important for low vision
forced-colors Windows High Contrast Mode ✅ Required for Windows users
hover Device supports hover interaction Important for touch vs mouse
pointer Device has precise pointing mechanism Important for touch vs mouse

Why this matters:

  • Respects user autonomy and accessibility needs
  • Reduces eye strain and fatigue
  • Improves overall user experience
  • Required by WCAG accessibility standards

Implementation Best Practices:

  1. Always implement prefers-color-scheme — it's now expected
  2. Use CSS variables to make theme switching easy
  3. Test with actual system settings enabled
  4. Don't override user preferences unless absolutely necessary
  5. Provide manual toggle for users who want to override system settings

Pro Tip: Combine multiple preferences for comprehensive accessibility:


/* Comprehensive theme system */
@media (prefers-color-scheme: dark) and (prefers-reduced-motion: reduce) {
    :root {
        --bg-color: #0a192f;
        --text-color: #e6f1ff;
        --accent-color: #ffb74d;
    }
    
    * {
        animation-duration: 0.01ms !important;

11.3 Responsive Design Patterns

Apply proven strategies for creating robust, maintainable responsive designs.

Breakpoint strategies

Choose meaningful breakpoints based on content, not devices.


/* Content-driven breakpoints */
@media (min-width: 480px) {
    /* Small mobile */
}

@media (min-width: 768px) {
    /* Tablet */
}

@media (min-width: 1024px) {
    /* Small desktop */
}

@media (min-width: 1200px) {
    /* Large desktop */
}

/* Device-based breakpoints (avoid) */
@media (min-width: 375px) { /* iPhone SE */ }
@media (min-width: 414px) { /* iPhone Plus */ }
@media (min-width: 768px) { /* iPad */ }
@media (min-width: 1024px) { /* iPad Pro */ }
@media (min-width: 1440px) { /* MacBook Air */ }

Responsive Breakpoints

Resize your browser to see how content adapts

Content-Driven vs Device-Driven:

Approach Pros Cons Recommendation
Content-Driven Based on actual content needs
Works on any device
Future-proof		 Requires testing
More thought required		 ✅ Recommended
Device-Driven Easy to implement
Matches popular devices		 Becomes outdated quickly
Not future-proof
Can break on new devices		 ❌ Avoid

Recommended Breakpoint Strategy:

  • Base (mobile): 0–479px
  • Small tablet: 480–767px
  • Large tablet: 768–1023px
  • Small desktop: 1024–1279px
  • Large desktop: 1280px+

How to determine breakpoints:

  1. Start with mobile view
  2. Gradually increase browser width
  3. At each point where content breaks or looks bad, add a breakpoint
  4. Don't guess — test with real content
  5. Use your actual content, not placeholders

Real-World Example:


/* Blog layout breakpoints */
.article {
    padding: 15px;
}

/* At 500px: Add horizontal padding */
@media (min-width: 500px) {
    .article {
        padding: 20px;
    }
}

/* At 700px: Two-column layout */
@media (min-width: 700px) {
    .article {
        display: grid;
        grid-template-columns: 2fr 1fr;
        gap: 30px;
    }
}

/* At 1000px: Larger typography */
@media (min-width: 1000px) {
    .article h1 {
        font-size: 2.5rem;
    }
}

Pro Tip: Use min-width queries consistently with mobile-first approach — it's more scalable and logical.

Fluid Grid Principles:

  • Use auto-fit and minmax() for truly responsive grids
  • Avoid fixed widths — use percentages or fr units
  • Use CSS Grid for complex layouts, Flexbox for simpler ones

  • Use object-fit: cover for constrained aspect ratios
  • Use aspect-ratio property to reserve space before image loads

Progressive enhancement approaches

Build a solid foundation and enhance it for capable devices — the philosophy of modern web development.


/* Base layer: Works everywhere */
.card {
    padding: 15px;
    border: 1px solid #ddd;
    border-radius: 5px;
    background-color: #fff;
}

/* Enhanced layer: Better visuals */
@media (min-width: 768px) {
    .card {
        padding: 20px;
        border-radius: 8px;
        box-shadow: 0 4px 8px rgba(0, 0, 0, 0.1);
    }
}

/* Advanced layer: Modern features */
@media (min-width: 1024px) and (hover: hover) {
    .card {
        border: 2px solid transparent;
        background-image: linear-gradient(#fff, #fff), 
                          linear-gradient(45deg, #00d1ff, #00a8ff);
        background-origin: border-box;
        background-clip: content-box, border-box;
    }
    
    .card:hover {
        background-position: 100% 100%, 0 0;
        background-size: 200% 200%;
        transition: background-position 0.3s ease;
    }
}

/* Experimental layer: Cutting-edge features */
@supports (mix-blend-mode: multiply) {
    .card {
        mix-blend-mode: multiply;
    }
}

Progressive Enhancement Layers:

  1. Base Layer: Core functionality that works everywhere
  2. Enhanced Layer: Improved UX for modern devices
  3. Advanced Layer: Cutting-edge features for latest browsers
  4. Experimental Layer: Future features with feature detection

  • Start with semantic HTML — content should be accessible without CSS

  • Don't rely on JavaScript for core functionality
  • Test on slow networks and old devices

@media (hover: hover) {
    .btn:hover {

Benefits of Progressive Enhancement:

  • Works on all devices and browsers
  • Fast loading on slow connections
  • Accessible to assistive technologies
  • Future-proof — adds features as they become available
  • Reduced maintenance costs

Pro Tip: Think of it as building a house — start with a strong foundation, then add rooms and decorations as resources allow.

Why component-based design matters:

  • Components are reusable anywhere — sidebar, main content, footer
  • No need for wrapper classes or context-dependent selectors
  • Identical components behave consistently regardless of context
  • Design systems become truly modular
  • Eliminates "desktop-only" bugs

Traditional Approach Problems:


/* Problem: Context-dependent styles */
.sidebar .card {
    /* Different styles for cards in sidebar */
}

.main-content .card {
    /* Different styles for cards in main content */
}

/* Hard to maintain, prone to errors */

Modern Solution:


/* Solution: Component self-containment */
.card {
    container-type: inline-size;
    container-name: card;
    /* All styling happens here */
}

@container card (min-width: 250px) {
    .card { /* Styles for larger contexts */ }
}

@container card (min-width: 350px) {
    .card { /* Styles for even larger contexts */ }
}

Pro Tip: Combine component-based design with container queries and CSS locks for the ultimate responsive component library.

Real-World Impact: This approach allows you to build a design system where every component adapts perfectly to its environment — whether it's in a narrow sidebar, a wide hero section, or a full-page modal.

11.4 Modern Responsive Techniques

Implement cutting-edge techniques for truly adaptive, intrinsic web design.

Intrinsic web design principles

Design based on content's natural dimensions rather than arbitrary breakpoints.


/* Intrinsic design example */
.article {
    max-width: 65ch; /* Natural reading width */
    margin: 0 auto;
    padding: 20px;
}

/* Typography based on content */
.heading {
    font-size: clamp(1.5rem, 5vw, 3rem);
    line-height: 1.2;
}

.paragraph {
    font-size: clamp(1rem, 3vw, 1.5rem);
    line-height: 1.6;
    margin-bottom: 1.6rem;
}

/* Grid based on content width */
.gallery {
    display: grid;
    grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
    gap: 20px;
}

/* Flexible images */
img {
    max-width: 100%;
    height: auto;
    aspect-ratio: attr(data-aspect-ratio);
}

Intrinsic Design Demo

This content adapts naturally to its container size, not arbitrary breakpoints. The text scales smoothly from mobile to desktop, always maintaining optimal readability.

Core Principles of Intrinsic Web Design:

  1. Content First: Design around what the content needs, not the device
  2. Fluidity: Allow elements to flow and resize naturally
  3. Constraints: Use minimum/maximum limits to prevent extremes
  4. Modularity: Components should adapt independently
  5. Typography: Let text size adjust based on container width

Contrast with Traditional Design:

Traditional Intrinsic
Fixed breakpoints Fluid scaling
Device-focused Content-focused
Manual overrides Automatic adaptation
Static layouts Dynamic layouts
One-size-fits-all Context-aware

Key Concepts:

  • max-width: 65ch — Optimal reading width for paragraphs
  • clamp() — Fluid typography with bounds
  • minmax() — Flexible track sizing
  • aspect-ratio — Preserve proportions
  • auto-fit — Adaptive grid columns

Pro Tip: Intrinsic design isn't about removing breakpoints — it's about using them wisely as fallbacks, not primary controls.


/* Intrinsic design with fallbacks */
.hero {
    display: grid;
    grid-template-columns: repeat(auto-fit, minmax(300px, 1fr));
    gap: 30px;
    max-width: 1200px;
    margin: 0 auto;
}

/* Fallback for older browsers */
@supports not (grid-template-columns: repeat(auto-fit, minmax(300px, 1fr))) {
    .hero {
        display: flex;
        flex-wrap: wrap;
    }
    
    .hero > * {
        flex: 1 1 300px;
    }
}
CSS locks and fluid typography

Create typography that scales smoothly between minimum and maximum sizes.


/* CSS Lock Formula */
.clamp-typography {
    font-size: clamp(min-size, preferred-size, max-size);
}

/* Practical examples */
.h1 {
    font-size: clamp(1.5rem, 5vw, 3rem);
}

.h2 {
    font-size: clamp(1.25rem, 4vw, 2.5rem);
}

.h3 {
    font-size: clamp(1.125rem, 3.5vw, 2rem);
}

.p {
    font-size: clamp(1rem, 3vw, 1.5rem);
}

/* More complex lock */
.complex-lock {
    font-size: clamp(
        16px,      /* Minimum: readable on mobile */
        4vw + 12px, /* Preferred: scales with viewport */
        48px       /* Maximum: prevents huge text on large screens */
    );
}

/* With custom properties */
:root {
    --font-size-base: 1rem;
    --font-size-scaling: 1.2;
}

.text {
    font-size: clamp(
        calc(var(--font-size-base) * 0.8),
        calc(var(--font-size-base) * var(--font-size-scaling)),
        calc(var(--font-size-base) * 2)
    );
}

CSS Lock Demo

This text scales smoothly from 16px on mobile to 24px on desktop, never going below 16px or above 24px.

Why CSS locks matter:

  • Eliminate the need for multiple media queries
  • Smooth transitions between sizes
  • Perfect for headlines and important text
  • Improves readability across all devices

Formula Components:

  • Minimum: Smallest readable size (typically 16px for body text)
  • Preferred: Scales with viewport — use vw units
  • Maximum: Largest acceptable size (prevents text from becoming too large)

Recommended Values:

Element Minimum Preferred Maximum
h1 1.5rem 5vw 3rem
h2 1.25rem 4vw 2.5rem
h3 1.125rem 3.5vw 2rem
p 1rem 3vw 1.5rem
caption 0.875rem 2.5vw 1.125rem

Pro Tip: Use clamp() for all typography — it's the modern standard for responsive text.

Advanced Technique: Combine with container queries for component-level fluid typography:


@container card (min-width: 300px) {
    .card-title {
        font-size: clamp(1.3rem, 4vw, 1.8rem);
    }
    
    .card-content {
        font-size: clamp(0.9rem, 2.5vw, 1.1rem);
    }
}

Art Direction Demo

Different images load based on device, orientation, and user preferences

What is Art Direction?

Changing the actual image content based on context — not just resizing it.

Traditional vs Art Direction:

Traditional Art Direction
Same image, resized Different image composition
Focus on central subject Reframe for context
Uniform across devices Context-aware composition
File size optimization Visual impact optimization

When to use Art Direction:

  • Hero banners — crop differently for mobile vs desktop
  • Product images — focus on different features
  • Portrait photos — reframe for vertical screens
  • Infographics — simplify for small screens
  • Dark/light mode — different color treatments

Picture Element Structure:

  1. <picture> — Container element
  2. <source> — Define conditions and sources
  3. <img> — Fallback for unsupported browsers

Source Attributes:

  • media: Media query condition
  • srcset: Image source URL
  • type: MIME type (for format selection)

Pro Tip: Always include a fallback <img> — it's required for accessibility and compatibility!

Modern Best Practice:


<picture>
    <source media="(max-width: 767px)" srcset="hero-mobile.avif" type="image/avif">
    <source media="(max-width: 767px)" srcset="hero-mobile.webp" type="image/webp">
    <source media="(max-width: 767px)" srcset="hero-mobile.jpg" type="image/jpeg">
    
    <source media="(min-width: 768px)" srcset="hero-desktop.avif" type="image/avif">
    <source media="(min-width: 768px)" srcset="hero-desktop.webp" type="image/webp">
    <source media="(min-width: 768px)" srcset="hero-desktop.jpg" type="image/jpeg">
    
    <img src="hero-default.jpg" alt="Hero image" loading="lazy">
</picture>

Performance Optimization:

  • Use modern formats: AVIF > WebP > JPEG
  • Always specify type attribute
  • Use loading="lazy" for below-the-fold images
  • Specify explicit width and height attributes
  • Use decoding="async" for better rendering

Accessibility Note: Always provide meaningful alt text — don't forget this critical requirement!
