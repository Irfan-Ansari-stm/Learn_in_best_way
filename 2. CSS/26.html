24.1 Email CSS Constraints

Email clients have wildly inconsistent CSS support. What works in Gmail fails in Outlook, and vice versa.

Definition: Supported Properties and Limitations

Email clients use different rendering engines:

  • Outlook (Windows): Microsoft Word engine (extremely limited)
  • Gmail: WebKit-based (good support, but strips some CSS)
  • Apple Mail: WebKit (excellent support)
  • Yahoo Mail: Modified WebKit
  • Mobile clients: Vary by device and app

Well-supported properties:

  • font-family, font-size, color
  • margin, padding, border
  • text-align, vertical-align
  • background-color
  • width, height
  • display: block, inline-block
  • line-height

Unsupported or unreliable properties:

  • CSS Flexbox and Grid
  • CSS animations and transitions
  • position: absolute/fixed
  • float
  • border-radius
  • box-shadow
  • background-image with gradients
  • media queries (limited support)

Pro tip: When in doubt, test it! Use tools like Litmus or Email on Acid.

Syntax: Inline Styles Necessity

Unlike web pages, email clients strip out <style> tags and external CSS files. You MUST use inline styles!

<!-- ‚ùå BAD: External or internal styles -->
<head>
  <style>
    .btn { background-color: #00d1ff; color: white; }
  </style>
</head>
<body>
  <a href="#" class="btn">Click Me</a>
</body>

<!-- ‚úÖ GOOD: Inline styles only -->
<body>
  <a href="#" style="background-color: #00d1ff; color: white; display: inline-block; padding: 10px 20px; text-decoration: none; border-radius: 4px;">Click Me</a>
</body>

Why this matters:

  • Most email clients ignore <style> tags entirely
  • Even when supported, they're stripped during spam filtering
  • Inline styles are the only reliable method

Use tools to automate this conversion:

  • Premailer: Converts CSS to inline styles
  • CSS-to-Inline-Styles: Node.js library
  • Mailchimp Template Editor: Built-in converter

24.2 Dark Mode and Email

Dark mode is now standard in email clients. Your emails must adapt gracefully to user preferences.

Definition: Email Dark Mode Support

Major email clients now support dark mode:

  • iOS Mail: Full dark mode support since iOS 13
  • macOS Mail: Full dark mode support
  • Gmail: Dark mode since 2019
  • Outlook (Mac): Supports dark mode
  • Outlook (Windows): Limited support

When dark mode is enabled, clients automatically invert colors unless you specify otherwise.

The problem: Automatic inversion often creates unreadable text!

Example: Blue text (#007bff) becomes orange (#ff7b00) ‚Äî completely wrong!

Syntax: Handling Dark Mode with CSS

There are two main approaches:

Method 1: CSS Custom Properties + Media Query

<head>
  <meta name="color-scheme" content="light dark">
  <meta name="supported-color-schemes" content="light dark">
  
  <style type="text/css">
    :root {
      --text-color: #333333;
      --bg-color: #ffffff;
      --link-color: #00d1ff;
      --button-bg: #00d1ff;
      --button-text: #ffffff;
    }
    
    @media (prefers-color-scheme: dark) {
      :root {
        --text-color: #e0e0e0;
        --bg-color: #121212;
        --link-color: #00d1ff;
        --button-bg: #00d1ff;
        --button-text: #000000;
      }
    }
  </style>
</head>

<body style="margin: 0; padding: 0; background-color: var(--bg-color); color: var(--text-color);">
  <table width="100%" cellpadding="0" cellspacing="0" border="0">
    <tr>
      <td align="center" style="background-color: var(--bg-color); color: var(--text-color);">
        <!-- Content -->
        <p style="font-family: Arial, sans-serif; font-size: 16px; color: var(--text-color);">
          This text adapts to dark mode!
        </p>
        
        <table cellpadding="0" cellspacing="0" border="0">
          <tr>
            <td style="background-color: var(--button-bg); border-radius: 4px;">
              <a href="#" style="display: inline-block; padding: 12px 24px; font-family: Arial, sans-serif; font-size: 16px; color: var(--button-text); text-decoration: none;">
                Call to Action
              </a>

        </table>
      </td>

  </table>
</body>

Method 2: CSS Color Adjustment

<!-- This forces specific colors regardless of theme -->
<style type="text/css">
@media (prefers-color-scheme: dark) {
  body {
    background-color: #121212 !important;
    color: #e0e0e0 !important;
  }
  
  .dark-mode-fix {
    background-color: #00d1ff !important;
    color: #000000 !important;
  }
  
  a {
    color: #00d1ff !important;
    text-decoration: underline !important;
  }
}
</style>

Method 3: Background Color Override

<!-- For clients that auto-invert colors -->
<div style="background-color: #ffffff; color: #333333;">
  <!-- Content -->
</div>

<!-- Force dark mode colors -->
<div style="background-color: #121212; color: #e0e0e0;">
  <!-- Content -->
</div>

<!-- Hide from light mode -->

</div>

<style type="text/css">

  }

    display: none !important;
  }
}
</style>

Detail: Image Considerations

Images behave differently in dark mode:

Problem: Auto-inversion

Some clients invert image colors in dark mode, making logos unreadable.

Solution: Force color preservation

<!-- Prevent dark mode inversion -->
<img src="logo.png" 
     alt="Logo" 
     style="filter: brightness(1) contrast(1); -webkit-filter: brightness(1) contrast(1);">

Best practices:

  • Use PNG images with transparent backgrounds
  • Avoid JPEG for logos (compression artifacts)
  • Test images in both light and dark modes
  • Provide alternative text for accessibility
  • Use width/height attributes for layout stability
  • Host images on secure servers (HTTPS)

Alternative approach: Two versions

<!-- Light mode version -->
<img src="logo-light.png" alt="Logo" class="light-only">

<!-- Dark mode version -->
<img src="logo-dark.png" alt="Logo" class="dark-only">
